## 2. 从问题反推：你真正想理解的是什么

从整组问题可以看出，你并不是只想记住某个知识点的定义，而是在持续追问以下几类“本质问题”。

### 2.1 代码最终是如何落到硬件上的
你关心的不只是 API 怎么调用，而是：
- 一行 C 代码最终变成了什么总线访问、寄存器读写、位操作或时序行为。
- “抽象接口”与“物理执行”之间到底隔了几层，每层分别做什么。

### 2.2 不同抽象层之间如何接力
你在反复追问：
- 用户态 / 内核态如何切换。
- Driver / HAL / Device Tree / Bus / Device / Driver 如何完成绑定。
- Bootloader / App / Startup / Linker Script / VTOR / 中断向量表如何完成运行时接力。

### 2.3 为什么这些机制必须这样设计
你不是只问“是什么”，而是在追问：
- 为什么 MMIO 比 I2C 更底层。
- 为什么 `.data` 会同时占用 Flash 和 SRAM。
- 为什么 RTOS 延时不是“死等”，而是“主动让权”。
- 为什么内核不能直接解引用用户指针，为什么要 `copy_from_user()`。
- 为什么 DMA、Cache、MPU、时钟树、硬件过滤器这些设计是必要的。

### 2.4 工程实践里真正容易踩坑的点
你特别关注“纸面知识”和“真实工程”之间的落差，例如：
- RMW 不是原子操作，会发生数据竞争。
- I2C 总线会挂死，SCL/SDA 可能出现异常半高、持续拉低。
- Bootloader 跳转前不清环境会 HardFault。
- `-Wl,` 传参、`.map` 文件、`--gc-sections` 这类工具链细节决定优化是否真正生效。
- DMA 与 Cache 一致性问题不是概念，而是会直接导致读写旧数据。

### 2.5 建立“一条线贯穿到底”的能力
你的问题整体上是在建立这样一条主线：

> 物理电路 / 引脚 / 总线  
> → 寄存器 / MMIO / 中断 / DMA  
> → 驱动抽象 / RTOS / Linux 内核机制  
> → 链接脚本 / 启动流程 / 内存布局  
> → 用户态接口 / 系统调用 / UI 联动  
> → 最终形成完整的软件-硬件闭环理解

这说明你真正想要的不是碎片知识，而是**从物理层到系统层的穿透式理解能力**。

## 3. 核心知识主线

### 3.1 硬件接口与底层通信本质
这一主线关注“电如何变成信息、信息如何变成控制”：
- MMIO 与协议通信的层级关系。
- I2C / SPI / UART / CAN 的物理时序和通信语义。
- GPIO 推挽 / 开漏 / 线与逻辑。
- Qi 无线充电的带内通信、本振调制、ASK / FSK 分工。
- ePWM 如何从时基、比较器、动作控制一直落到功率驱动。

你想理解的是：**所谓“通信协议”在硬件里究竟靠什么实现，它和直接访问寄存器相比差在哪。**

### 3.2 CPU、内存与寄存器访问模型
这一主线关注“处理器到底在访问什么”：
- SRAM、寄存器、Flash、外设空间的角色差异。
- 绝对地址、结构体映射、`volatile` 的真实意义。
- RMW 的底层三步动作与非原子风险。
- 共享内存、硬件信号量、位带操作等并发控制手段。

你想理解的是：**地址不是抽象符号，而是总线交易入口；位操作不是语法糖，而是硬件状态修改。**

### 3.3 驱动架构与设备模型
这一主线关注“统一接口背后如何绑定到具体硬件”：
- HAL 与 Driver 的分层哲学。
- 用函数指针表实现底层多态。
- Bus / Device / Driver 三张表如何解耦。
- Device Tree 如何把“硬件参数”从“驱动代码”中分离出去。
- Probe、注册、匹配、运行时调用形成怎样的闭环。

你想理解的是：**驱动不是“调库”，而是“抽象接口 + 运行时绑定 + 底层实现”的整体系统。**

### 3.4 并发、调度与执行流切换
这一主线关注“为什么系统能同时做很多事”：
- 线程、进程、LWP 的关系。
- 时间片轮转如何制造“并发幻觉”。
- RTOS 的 SysTick / PendSV / 上下文切换。
- Linux CFS、红黑树、`vruntime` 的公平调度。
- Block、Mutex、Atomic、自旋锁、信号量的适用边界。

你想理解的是：**并发不是同时执行那么简单，而是“执行流切换 + 状态保存 + 共享资源保护”的一整套机制。**

### 3.5 存储布局、链接脚本与启动流程
这一主线关注“程序在上电后如何真正跑起来”：
- `.text` / `.data` / `.bss` / Heap / Stack 的职责。
- LMA 与 VMA 的双地址模型。
- `Startup.s` 在 `main()` 之前完成哪些搬运与清零动作。
- `section` 属性和链接脚本如何把代码 / 数据放进指定物理区域。
- Bootloader 与 App 如何通过分区、VTOR、MSP 校验和环境清理实现跳转。

你想理解的是：**程序不是“编译完就能跑”，而是要经过链接、装载、重定位、初始化、向量接力后才真正进入运行态。**

### 3.6 Linux 内核与用户态的边界
这一主线关注“权限和地址空间如何塑造系统结构”：
- Ring 0 / Ring 3 的权限差异。
- 用户虚拟地址、内核虚拟地址、页表与 MMU。
- `write()` / `ioctl()` 如何通过 Trap / SVC 进入内核。
- 为什么要 `copy_from_user()` / `copy_to_user()`。
- `malloc` / `calloc` / `kzalloc` 的语义差别与安全意义。

你想理解的是：**用户态和内核态并不是“两个库”，而是两套权限和地址空间规则下的执行世界。**

### 3.7 工具链与优化落地
这一主线关注“优化是否真正发生”：
- `.hex` 与 `.bin` 的差异。
- `-ffunction-sections`、`-fdata-sections`、`--gc-sections` 的配合。
- `.map` 文件如何验证链接结果。
- 对齐、位域、段裁剪、编译优化等级如何影响体积和性能。

你想理解的是：**优化不是口号，而是要从编译器、链接器、内存布局和结果验证四个层面一起落地。**

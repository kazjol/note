# 设备树与 Probe / Match 流程

## 1. 这篇文档解决什么问题

本专题回答的是：

> **Linux 驱动里常说的“设备树描述硬件，驱动通过 compatible 匹配，probe 时完成绑定”到底是一条怎样的完整链路？**

很多人知道这些名词，但真正一到工程里就会混乱：
- 设备树到底只是“配置文件”还是“硬件数据库”？
- compatible 是字符串而已，为什么能决定驱动绑定？
- probe 为什么有时进、有时不进、有时会延迟？
- 设备树、platform device、driver、sysfs 之间的关系到底是什么？

这篇文档就是专门把这条链梳理顺。

---

## 2. 核心结论先看

### 2.1 设备树不是驱动代码，而是“硬件描述数据”
设备树的核心职责是告诉内核：
- 这里有什么硬件；
- 它挂在哪；
- 基地址是多少；
- 中断号、DMA、时钟、GPIO、电源等资源是什么。

它本身不做驱动逻辑。

### 2.2 驱动匹配本质是“设备声明自己是谁，驱动声明自己支持谁”
- 设备树节点里的 `compatible` 表达“这个硬件是什么 / 兼容谁”。
- 驱动里的 `of_device_id` 表表达“我支持哪些 compatible”。

两边对上了，匹配才成立。

### 2.3 probe 不是“系统一启动自动跑一次普通函数”那么简单
probe 只有在：
- 设备对象已经被注册出来；
- 驱动对象也已经注册；
- 匹配关系成立；
- 资源依赖基本满足；

时才会发生。

所以 probe 是**设备模型运行时绑定点**，不是一个孤立 API。

---

## 3. 为什么 Linux 要把“硬件参数”和“驱动逻辑”拆开

在很早期或很简单的系统里，驱动可能把这些东西写死：
- 基地址；
- 中断号；
- GPIO 号；
- 板级差异。

这种写法的问题是：
- 同一驱动适配不同板子会变得痛苦；
- 板级差异和驱动逻辑混在一起；
- 每次换板可能都要改代码重编。

设备树出现后，思路变成：
- 驱动负责“怎么驱”；
- 设备树负责“这块板上的硬件具体在哪、资源是什么”。

这就是典型的：

> **逻辑与硬件参数解耦。**

---

## 4. 从设备树到 platform device：设备对象是怎么来的

## 4.1 设备树节点本身不是驱动对象
设备树里的一个节点，例如：
```dts
uart0: serial@10020000 {
    compatible = "vendor,uart-v1";
    reg = <0x10020000 0x1000>;
    interrupts = <5>;
    clocks = <&clk_uart0>;
    status = "okay";
};
```

它首先只是**一段描述数据**。

### 4.2 内核如何把它变成“设备实例”
启动过程中，内核会解析设备树，把满足条件的节点转成设备模型中的对象。
对于很多 SoC 内部固定外设，这通常会体现为：
- 创建 `platform_device`；
- 挂到 `platform_bus`；
- 携带节点里的资源信息。

所以 platform device 的作用就是：

> **把“设备树中的硬件描述”落成“内核里的设备实例对象”。**

---

## 5. 驱动注册后发生了什么

一个典型 platform 驱动会提供：
- `probe`
- `remove`
- `driver.name`
- `of_match_table`

例如它会声明自己支持：
```c
static const struct of_device_id mydrv_of_match[] = {
    { .compatible = "vendor,uart-v1" },
    { }
};
```

当驱动注册到 `platform_bus` 后，内核就会尝试：
- 在 bus 上已有设备里找能匹配的；
- 或未来新出现设备时再尝试匹配。

所以匹配并不一定只有一个时间点，它是一个“设备和驱动双向到位就尝试绑定”的过程。

---

## 6. Match 到底如何成立

## 6.1 compatible 的本质
`compatible` 不是随便写的注释字符串，而是：
- 设备身份标签；
- 兼容层级声明；
- 驱动选择入口。

例如：
```dts
compatible = "vendor,soc-uart-v2", "vendor,soc-uart-v1";
```

这表示：
- 优先声明自己是 v2；
- 同时兼容 v1 驱动语义。

### 为什么这很重要
这样可以让：
- 新硬件尽量匹配新驱动；
- 老驱动在兼容条件下仍可工作；
- 平台演进时减少驱动碎裂。

## 6.2 匹配不是只有一种机制
在 Linux 里，不同总线可以有不同匹配方式：
- OF compatible
- ACPI
- ID table
- name

但在你当前关注的 SoC / platform 设备语境里，最典型的就是：
- 设备树 compatible
- 驱动 of_match_table

## 7. Probe 里到底应该做什么

可以把 probe 理解成：

> **“设备对象和驱动实现已经正式绑定，现在开始把这块硬件真正接起来。”**

典型动作包括：
- 获取 `reg` 对应的寄存器资源；
- `ioremap` 或平台等价方式映射寄存器；
- 获取中断号；
- 获取并打开时钟；
- 获取 reset / regulator / pinctrl / GPIO；
- 初始化硬件寄存器；
- 申请 IRQ；
- 注册字符设备 / netdev / input 设备 / 子系统对象；
- 在 `drvdata` 中保存上下文。

### 一句话理解
probe 不是“识别设备”，而是：
- **识别已经结束后，开始完成资源接线和功能初始化。**

---

## 8. 为什么 probe 会失败或延迟

## 8.1 资源没准备好
例如：
- 时钟驱动还没就绪；
- regulator 没准备好；
- pinctrl 依赖没到位；
- 上游父设备还没准备好。

这时 probe 可能返回“稍后再试”类状态。

### 为什么这很常见
Linux 启动时并不是所有驱动和设备一次性按理想顺序到位，所以依赖管理非常重要。

## 8.2 设备树写错
常见错误：
- compatible 不匹配；
- `status = "disabled"`；
- `reg` 长度 / 地址错误；
- 中断号错；
- 时钟 phandle 错；
- 缺少必须属性。

### 表现
- probe 根本不进；
- probe 进去后申请资源失败；
- 驱动看起来加载了，但硬件完全不工作。

---

## 9. `status = "okay"` 为什么重要

很多设备树节点默认并不一定启用。
如果 `status` 没写对，或者是 `disabled`，即使：
- 驱动在；
- compatible 也对；

设备对象也可能根本不会进入可绑定状态。

所以“probe 不进”时，设备树里首先要看：
- 节点是否存在；
- `status` 是否真的启用。

---

## 10. 从设备树资源到驱动 API 的映射直觉

你可以把设备树资源理解成“驱动启动所需的资源清单”：
- `reg` → 寄存器地址窗口
- `interrupts` → 中断来源
- `clocks` → 时钟依赖
- `resets` → 复位控制
- `gpios` → 外围引脚控制
- `dmas` → DMA 通道
- `pinctrl-*` → 引脚复用状态
- `power-domains` / regulator → 供电依赖

probe 过程本质上就是把这份“资源清单”变成真正可用的内核资源句柄。

---

## 11. sysfs 和驱动绑定关系怎么看

当设备与驱动成功绑定后，通常可以在 sysfs 中看到：
- 设备节点；
- 它挂在哪个 bus；
- 它绑定到了哪个 driver；
- 某些资源和属性。

所以 sysfs 是观察：
- 设备有没有被创建设备对象；
- 驱动有没有真的绑定上；
- 绑定关系是否正确；

的很好入口。

这也是为什么很多 Linux 驱动排障都离不开：
- `dmesg`
- `/sys`
- 驱动 probe 日志

---

## 12. 一条完整闭环链

把整个过程串起来，就是：

> 设备树描述硬件  
> → 内核解析节点  
> → 创建设备对象（如 platform_device）  
> → 驱动注册到对应 bus  
> → compatible / match 成功  
> → probe 获取资源并初始化硬件  
> → 注册到上层子系统 / 提供用户可见接口

只要这条链清楚，设备树和 probe 就不再是零散名词，而是一整套可追踪流程。

---

## 13. 最容易踩坑的点
- 把设备树当成“驱动代码的一部分”而不是独立硬件描述。
- 以为 compatible 只是备注字符串。
- probe 不进时，只盯驱动，不查设备树和 bus 上是否真的有设备对象。
- 忽视 `status`、时钟、reset、pinctrl 等依赖资源。
- 不理解 probe 是绑定后初始化，而不是“发现设备”的唯一动作。

---

## 14. 调试 probe 不进时的排查顺序

建议按这个顺序查：
1. 设备树节点是否存在。
2. `status` 是否为 `okay`。
3. compatible 是否和驱动的 match table 对得上。
4. 设备对象是否真的创建到了对应 bus。
5. 驱动是否成功注册。
6. probe 是否因依赖没就绪而 defer。
7. 资源属性（reg / irq / clocks / reset / gpio）是否正确。

这个顺序通常能快速定位卡点。

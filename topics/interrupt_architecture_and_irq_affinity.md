# 中断架构与 IRQ 亲和性

## 1. 这篇文档解决什么问题

本专题回答的是：

> **中断为什么能让 CPU 立刻响应外部事件？它在系统里到底是如何分层组织的？在多核系统里又为什么要考虑 IRQ 亲和性？**

它重点解决这些问题：
- 中断不是“函数回调”，而是硬件打断当前执行流的机制。
- 中断控制器、设备、中断号、CPU 核之间是什么关系。
- 为什么中断处理必须分轻重层次，不能把所有逻辑都塞进 ISR。
- 多核系统为什么要做中断绑核，以及这和性能、Cache 有什么关系。

---

## 2. 核心结论先看

### 2.1 中断的本质是“异步抢占”
CPU 原本正在执行当前代码流，但某个外设或事件到来时，会通过中断机制：
- 暂停当前执行流；
- 保存必要现场；
- 跳到预设中断入口；
- 执行 ISR；
- 再返回原先流程或触发新的调度。

所以中断不是普通函数调用，而是由硬件和异常机制共同驱动的执行流切换。

### 2.2 中断架构是分层的
真实系统里的中断不是“设备直接敲 CPU 门”，中间通常还会经过：
- 外设本地中断源；
- 中断控制器；
- 层级路由；
- CPU 内核异常入口。

这就是为什么会有 `interrupt-parent` 这类描述层级的属性。

### 2.3 中断处理应该“上半部快，下半部重活少做”
ISR 应尽量只做：
- 读状态；
- 清中断；
- 搬最必要的数据；
- 唤醒后续处理。

如果把大量逻辑、阻塞操作、复杂协议解析都塞进 ISR，系统会很容易变差甚至失控。

### 2.4 IRQ 亲和性本质上是“把中断偏向某个核心处理”
在多核系统里，中断不一定必须由所有核心平均承担。
有些场景会主动把特定 IRQ 固定或偏向某个核心，以：
- 减少迁移；
- 降低 cache 污染；
- 避免高优先任务被无关中断打扰；
- 做性能隔离。

---

## 3. 中断从设备到 CPU 的完整路径

## 3.1 外设先产生事件
中断最初通常来自某个硬件事件，例如：
- UART 收到数据；
- DMA 完成搬运；
- 定时器到期；
- GPIO 边沿触发；
- I2C 传输完成或出错。

外设内部会设置自己的状态位，并把中断请求往上游送。

## 3.2 中断控制器负责仲裁与分发
中断控制器的职责通常包括：
- 汇总多个中断源；
- 判断优先级；
- 决定是否屏蔽；
- 把中断送给 CPU。

在 Cortex-M 里常见的是 NVIC；
在更复杂的 SoC / Linux 系统里常见的是 GIC 或更复杂层级。

## 3.3 CPU 进入异常入口
当 CPU 接收中断并决定响应后，会：
- 保存必要上下文；
- 跳入中断向量表对应入口；
- 执行 ISR。

这一步和普通函数调用最大的区别是：
- 它不是由当前代码显式 `call` 出去的；
- 而是由硬件异步打断当前执行流。

---

## 4. 为什么中断一定要有优先级

因为多个事件可能同时发生，而 CPU 的响应能力是有限的。

优先级的意义在于：
- 更紧急的事件先处理；
- 某些中断可以打断另一些较低优先级中断；
- 保证关键实时链路的响应上限。

但优先级也不是越高越好。
如果过多中断都设太高，会导致：
- 抢占层级混乱；
- 系统难分析；
- 低优先任务长期得不到运行机会。

所以好的中断设计不是“全设最高”，而是按实时性和重要性合理分层。

---

## 5. 中断上下文和进程 / 任务上下文的边界

## 5.1 中断上下文的核心特点
中断上下文通常意味着：
- 响应要快；
- 不应长时间占用 CPU；
- 不能随意阻塞；
- 适合做最小必要动作。

## 5.2 为什么不能在 ISR 里做重活
如果 ISR 太重，会导致：
- 更高优先级事件被延迟；
- 系统抖动变大；
- 中断嵌套复杂度提升；
- 调试困难；
- 实时性下降。

特别是在 Linux 里，还需要更严格地区分：
- 硬中断；
- 软中断 / tasklet / workqueue 等后续处理机制。

虽然实现细节不同，但核心思想是一致的：

> **中断入口只做快事，重活尽量后移。**

### 5.3 在 MCU / RTOS 中的典型分工
- ISR：清标志、读寄存器、取少量数据、发信号。
- 任务：解析协议、打印日志、状态机处理、UI / 业务逻辑。

这也是把中断和 RTOS 调度配合好的关键。

---

## 6. `interrupt-parent` 与中断层级“套娃”

在复杂 SoC 里，一个设备的中断不一定直接连到 CPU，常常是：
- 设备 → 子中断控制器 → 主中断控制器 → CPU

所以设备树里需要 `interrupt-parent` 之类的属性来说明：
- 这个设备的中断最终挂在哪个控制器下面；
- 中断号该如何解释；
- 触发类型和层级关系是什么。

这也是你问到“套娃结构”的本质。

理解这一点后，你就会明白：
- 中断号不是抽象 magic number；
- 它总是依附于某一层中断域解释。

---

## 7. 中断与 DMA 的关系

DMA 常常会让人误以为“不需要 CPU 了”。
但真实情况是：
- DMA 负责搬数据；
- 搬完后仍通常要通过中断通知 CPU。

所以一条典型链路是：
- 外设触发 DMA；
- DMA 完成传输；
- DMA 中断到来；
- ISR 收尾并唤醒上层处理。

这说明 DMA 并不是取代中断，而是与中断形成配合：
- DMA 减少搬运负担；
- 中断负责通知和状态接力。

## 8. Linux 里的 IRQ 亲和性到底在解决什么

## 8.1 什么是 IRQ 亲和性
IRQ 亲和性就是：
- 指定或限制某个中断更倾向在哪些 CPU 核上被处理。

在 Linux 中常通过相关接口或 `smp_affinity` 掩码控制。

## 8.2 为什么要绑核
如果一个中断总在不同核心间漂移，会带来这些代价：
- Cache 热数据在不同核间不复用；
- 某些工作线程和中断处理不在同一核，局部性变差；
- 高负载业务核被额外打扰；
- 延迟抖动增大。

把特定中断偏向某个核的好处是：
- 数据和处理逻辑更容易留在同一核 cache；
- 某些关键任务核可以更“干净”；
- 整体时延更可预测。

### 8.3 典型场景
- 网络收包中断与对应处理线程放在相近核心；
- AI / 推理主核尽量少受无关设备中断干扰；
- 音频、视频、通信等实时链路固定某些 IRQ 路径，降低抖动。

---

## 9. 中断风暴与性能问题

如果系统没有合理设计中断路径，会出现：
- 中断太频繁，CPU 大量时间花在 ISR；
- 低优先级任务几乎跑不到；
- cache 被打爆；
- 延迟飘忽不定；
- 实际业务吞吐反而下降。

这也是为什么：
- CAN 要用硬件过滤器；
- DMA 要减少逐字节中断；
- 某些外设要做中断合并或批处理；
- Linux 平台要调亲和性、NAPI、轮询等机制。

本质上都在解决同一个问题：

> **不要让 CPU 因为“响应太积极”反而忙死。**

---

## 10. 中断设计的实用原则

### 10.1 先区分“必须立刻做”和“可以稍后做”
必须立刻做的：
- 清硬件标志；
- 取走会丢失的数据；
- 停止进一步灾难扩散。

可以稍后做的：
- 复杂协议解析；
- 打印；
- UI 更新；
- 大块数据处理；
- 长时间计算。

### 10.2 控制中断频率
优先考虑：
- 硬件过滤；
- DMA；
- 批量处理；
- 合理阈值；
- 不要让每个字节都打一次重中断。

### 10.3 把中断看成“最快的通知链路”，不是“万能业务入口”
这条原则非常重要。中断最适合：
- 快速通知；
- 最小收尾；
- 唤醒后续处理。

不适合承担完整业务流。

---

## 11. 本专题最容易踩坑的点
- 把中断当成普通回调函数理解。
- 在 ISR 里放过多业务逻辑。
- 不理解中断控制器与中断号的层级归属。
- 低估 DMA 完成中断、GPIO 中断、定时器中断叠加后的系统压力。
- 忽视中断亲和性对 cache 和时延的影响。
- 把“中断多”误解成“系统响应更好”。

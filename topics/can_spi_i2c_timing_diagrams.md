# CAN、SPI、I2C 时序图解

## 1. 这篇文档解决什么问题

本专题回答的是：

> **为什么协议文档里总是有一堆时序图，这些高低电平和采样点到底对应什么实际动作？**

它重点帮助你建立一个能力：
- 看到时序图，不只是“知道线在变”，而是真能读出：
  - 谁在驱动；
  - 谁在采样；
  - 当前阶段在传什么；
  - 错误会出现在哪。

本篇选取你最关心的三类总线：
- I2C
- SPI
- CAN

并尽量用“图 + 解释 + 坑点”的方式整理。

---

## 2. 看时序图时最先要问的 4 个问题

不管是什么协议，先问这四件事：
1. **哪根线负责时序基准？**
2. **哪一方在主动驱动当前信号？**
3. **数据是在什么时候稳定，什么时候被采样？**
4. **帧的开始、结束、应答、错误分别靠什么表示？**

只要这四点清楚，很多协议图就不再难读。

---

# 3. I2C 时序图解

## 3.1 I2C 最核心的直觉
I2C 是：
- 两根线：SCL、SDA；
- 开漏 + 上拉；
- 共享总线；
- 由主机发起时序。

### 最重要规则
**当 SCL 为高时，SDA 应保持稳定；SDA 在 SCL 高期间发生跳变，通常表示特殊条件。**

这条规则几乎是理解 I2C 时序的总钥匙。

---

## 3.2 起始条件（Start Condition）

```text
SCL  ────────────────┐────────
SDA  ────────┐_______│________
             ↓
        SDA 在 SCL 为高时由高变低
```

### 含义
这表示：
- 主机宣布“我要开始一次事务了”；
- 总线上所有从机都应准备进入地址 / 数据接收状态。

### 为什么它特殊
因为正常数据位变化通常发生在 SCL 低电平期间，而这里是在 SCL 高电平时改变 SDA，所以它不会被误当成普通数据位。

---

## 3.3 停止条件（Stop Condition）

```text
SCL  ────────────────┐────────
SDA  ________┌───────│────────
             ↑
        SDA 在 SCL 为高时由低变高
```

### 含义
这表示：
- 当前 I2C 事务结束；
- 总线被释放，回到空闲状态。

---

## 3.4 发送一个字节 + ACK

```text
SCL   _|-|_|-|_|-|_|-|_|-|_|-|_|-|_|-|_|-
SDA   <b7><b6><b5><b4><b3><b2><b1><b0><ACK>
       1   0   1   0   1   1   0   0    0
```

### 理解方法
- 前 8 个时钟：传 8 个 bit；
- 第 9 个时钟：应答位；
- 发送方在前 8 位驱动 SDA；
- 接收方在第 9 位负责拉低 SDA 表示 ACK。

### ACK 是什么
ACK = 0，表示：
- “我收到了”；
- “这个字节有效”；
- “可以继续传下一个字节”。

如果第 9 位没被拉低，就是 NACK。

---

## 3.5 地址 + 读写位

I2C 事务通常不是一上来就传纯数据，而是先发：
- 7 位从机地址；
- 1 位 R/W；
- 然后 ACK。

例如：
```text
[ A6 A5 A4 A3 A2 A1 A0 ][R/W][ACK]
```

### 含义
- `R/W = 0`：主机写从机。
- `R/W = 1`：主机读从机。

所以 I2C 的第一字节本质上是“寻址 + 方向声明”。

---

## 3.6 重复起始（Repeated Start）

```text
Start → Addr+W → ACK → RegAddr → ACK → ReStart → Addr+R → ACK → Data
```

### 为什么需要它
很多寄存器型从机读流程是：
1. 先写寄存器地址；
2. 不释放总线；
3. 再重新起始并切换成读。

这样可以保持事务连续性，避免别的主机插入。

---

## 3.7 I2C 最常见的时序坑点
- 忘记第 9 位 ACK 的方向是接收方控制。
- 不理解 Start / Stop 是在 SCL 高电平期间识别的。
- 看到 SDA 卡低却不知道可能是从机残留状态。
- 忽略时钟拉伸，以为 SCL 被拉住一定是故障。
- 上升沿过慢导致采样点读错。

---

# 4. SPI 时序图解

## 4.1 SPI 最核心的直觉
SPI 一般有：
- SCK：时钟；
- MOSI：主发从收；
- MISO：从发主收；
- CS / NSS：片选。

它的本质是：

> **一边打时钟，一边两端移位寄存器同步交换数据。**

所以 SPI 天生适合高速、全双工、短距离板级通信。

---

## 4.2 一次最基本的 SPI 事务

```text
CS    ________|-----------------------|_______
SCK   ________|-|_|-|_|-|_|-|_|-|_|-|_|-|____
MOSI  ________<b7><b6><b5><b4><b3><b2><b1><b0>
MISO  ________<q7><q6><q5><q4><q3><q2><q1><q0>
```

### 含义
- CS 拉低：选中从设备；
- 每来一个 SCK 脉冲：双方各移出 1 bit；
- MOSI 和 MISO 同时进行；
- CS 拉高：本次事务结束。

### 最关键理解
SPI 不是“先发再收”，而是：
- **发和收同时发生。**

所以你问过的那个点非常关键：

> **SPI 想读数据，也必须继续发时钟。**

通常主机会发 Dummy Byte（如 `0xFF` / `0x00`）来把从机数据“挤出来”。

---

## 4.3 CPOL / CPHA 是什么

SPI 时序最常见的困扰来自这两个参数：
- CPOL：空闲时钟电平。
- CPHA：在哪个边沿采样 / 改变数据。

### 理解核心
你可以不用一开始死记 4 种模式编号，而是先抓住：
- **空闲时钟是高还是低？**
- **数据是在第一个边沿采样还是第二个边沿采样？**

只要主从两端对这两点一致，通信就能对上。

### 典型后果
如果 CPOL / CPHA 配错：
- 数据整体左移 / 右移一位；
- 每个字节都错；
- 某些位看起来偶发正确，容易误导排查。

---

## 4.4 SPI 读寄存器的直觉流程
很多 SPI 设备会这样设计：
1. 主机先发“读命令 + 地址”；
2. 从机收到后准备数据；
3. 主机再继续发 Dummy Byte；
4. 从机在 MISO 上吐出目标数据。

所以波形上会看到：
- 前一段是命令阶段；
- 后一段才是有效返回数据阶段。

---

## 4.5 SPI 最常见的时序坑点
- 以为“不发数据也能收数据”。
- 忽视 CS 的拉低 / 拉高边界语义。
- 主从 CPOL / CPHA 不匹配。
- 字节序、bit 序（MSB first / LSB first）弄错。
- 从机准备时间不够，主机时钟给太快。

---

# 5. CAN 时序图解

## 5.1 CAN 最核心的直觉
CAN 和 I2C / SPI 很不一样，它不是简单的主从点对点时序，而是：
- 多节点共享总线；
- 差分信号；
- 广播式；
- 带仲裁。

它最独特的地方是：

> **多个节点可以同时开始发，但优先级高的帧会在仲裁中赢下来，而且不会像普通冲突那样把总线彻底撞坏。**

---

## 5.2 Dominant / Recessive 的直觉
CAN 总线上有两种逻辑状态：
- Dominant（显性）
- Recessive（隐性）

可以把它粗略理解成：
- 显性位更“强”；
- 只要有人发显性，总线就体现为显性；
- 只有所有人都释放，总线才体现为隐性。

这和 I2C 的线与逻辑在理解上有点像，但物理实现是差分总线，不是简单开漏单端线。

---

## 5.3 CAN 帧的大致结构

```text
SOF | Arbitration | Control | Data | CRC | ACK | EOF
```

### 重点理解
- **SOF**：帧开始；
- **Arbitration**：决定谁赢得总线；
- **Data**：有效载荷；
- **CRC**：校验；
- **ACK**：接收确认；
- **EOF**：帧结束。

所以 CAN 不只是“发 ID + 数据”，它本身就是一个完整的总线仲裁与可靠通信机制。

---

## 5.4 仲裁为什么这么巧妙
假设两个节点同时发：
- 某一位上，一个节点发隐性；
- 另一个节点发显性；
- 总线结果表现为显性；
- 发隐性的节点检测到“我发的和总线不一样”，就知道自己输了，停止发送。

### 关键点
因为仲裁位期间大家都在边发边听，所以能做到：
- 高优先级帧先发；
- 低优先级帧自动退让；
- 总线不会像普通冲突那样整帧报废。

这就是 CAN 非常经典也非常优雅的设计。

---

## 5.5 为什么 CAN 必须重视过滤器
CAN 是广播式的。
也就是说：
- 总线上的每一帧，理论上所有节点都能看到；
- 但 CPU 不应该真的处理所有帧。

所以控制器里通常会有**硬件过滤器**，让无关 ID 在硬件层就被拦下，避免 CPU 被中断烦死。

这个点在工程上非常关键。

---

## 5.6 CAN 最常见的时序 / 机制坑点
- 不理解仲裁是边发边比较总线。
- 分不清显性 / 隐性的优先关系。
- 忽视位时序配置（采样点、时间段、同步段）对稳定性的影响。
- 不配置过滤器，导致 CPU 被广播流量打爆。
- 把 CAN 当作“带地址的 UART”理解，这会严重低估它的总线机制复杂度。

---

# 6. 三种协议横向对比

## 6.1 I2C
- 两线；
- 开漏；
- 半双工语义更强；
- 有地址；
- 主机主导时序。

## 6.2 SPI
- 多线；
- 时钟与数据分离；
- 全双工；
- 通常无总线寻址，靠片选；
- 速度高、协议简单但板级线更多。

## 6.3 CAN
- 差分；
- 广播；
- 带仲裁；
- 可靠性高；
- 总线层机制复杂，适合多节点分布式控制。

---

# 7. 读时序图的通用技巧

看到时序图时，建议按这个顺序读：
1. 先看空闲状态是什么。
2. 再看开始条件靠什么定义。
3. 再找采样边沿。
4. 再分清谁在驱动哪根线。
5. 最后看 ACK / CRC / 仲裁 / 停止等“控制语义”。

这样读图效率会高很多。

---

## 8. 本专题最容易踩坑的点
- 把所有时序图都当“线电平变化示意”，不去对应真实协议语义。
- I2C 忽视 Start / Stop 的识别条件。
- SPI 不理解“读也要发”。
- CAN 不理解仲裁与广播机制。
- 只会背定义，不会把波形和硬件行为对应起来。

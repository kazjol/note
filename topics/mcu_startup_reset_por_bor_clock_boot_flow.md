# MCU 上电、复位、POR / BOR、时钟与启动链路

## 1. 这篇文档解决什么问题

本专题回答的是：

> **一颗 MCU 从“刚通电、还什么都没跑”到“开始执行你的程序”，中间到底经历了哪些硬件和软件步骤？POR、BOR、复位源、时钟树、启动代码之间又是什么关系？**

这正是很多底层问题真正的起点。你前面关心的很多现象，例如：
- 为什么有时刚上电和手动复位行为不一样；
- 为什么电压不稳时会反复重启；
- 为什么某些时钟切换后程序异常；
- 为什么 `main()` 前已经发生了很多事；

本质上都和这条启动链有关。

---

## 2. 核心结论先看

### 2.1 MCU 不是“上电就直接进 main()”
真实过程通常是：
- 电源达到允许工作范围；
- 触发并释放上电复位；
- 复位逻辑把系统拉回初始状态；
- CPU 从启动地址取向量表；
- 先执行启动代码；
- 初始化数据段 / 时钟 / C 运行时；
- 最后才进入 `main()`。

### 2.2 POR 和 BOR 解决的是“电压不健康时不要乱跑”
- **POR（Power-On Reset）**：解决上电过程电压还没稳定时，芯片不能乱执行。
- **BOR（Brown-Out Reset）**：解决运行中电压跌落到不安全区时，芯片不能继续在半死不活状态下执行。

这两个机制本质上都是：

> **当供电不可靠时，强制系统保持或回到复位态。**

### 2.3 时钟是启动链的一部分，不是启动后的附属配置
CPU 能否正常跑、外设能否工作、总线节拍是否正确，都依赖时钟。
所以启动链里一定会涉及：
- 默认时钟源；
- 后续是否切到 HSE / PLL；
- Flash 等待周期是否匹配；
- 外设时钟是否开启。

---

## 3. 什么是 POR（Power-On Reset）

## 3.1 POR 为什么存在
当芯片刚接上电源时，电压不会在 0 和目标值之间瞬间跳变到位，而是一个上升过程。
在这个过程中，如果 MCU 太早开始执行，可能会出现：
- 某些逻辑单元已工作，某些还没稳定；
- 时钟源还不可靠；
- SRAM / 寄存器状态不可信；
- CPU 取指和总线行为混乱。

所以 POR 的作用是：
- 在供电还没达到安全门槛前，强制保持复位；
- 只有当电源达到可用范围后，才允许启动。

### 3.2 直觉理解
你可以把 POR 理解成：

> **“电还没站稳，先别动。”**

---

## 4. 什么是 BOR（Brown-Out Reset）

## 4.1 BOR 为什么存在
即使系统已经正常运行，供电也可能因为：
- 电池电量低；
- 瞬时负载过大；
- 电源抖动；
- 热插拔 / 线缆问题；
- LDO / DCDC 瞬态响应不足；

导致电压跌到不安全区。

如果此时 MCU 继续执行，就可能出现：
- 指令乱跑；
- Flash 读写错误；
- SRAM / 寄存器内容异常；
- 外设状态机失控；
- 正在写 Flash 时损坏数据。

所以 BOR 的意义是：
- 一旦电压跌过阈值，就强制复位；
- 等电压恢复到安全范围后再重新启动。

### 4.2 直觉理解
如果 POR 是“别太早起床”，那 BOR 就是：

> **“你已经在工作了，但现在供电不行了，立刻停下来重来。”**

---

## 5. POR 和 BOR 的差别

可以这样区分：

### POR
- 关注上电过程；
- 目的是防止芯片在电压爬升阶段乱启动。

### BOR
- 关注运行中掉压；
- 目的是防止芯片在低压不稳状态下继续执行。

很多实际芯片里，这两套机制会共同构成电源监控与复位体系。

---

## 6. 复位不只有一种来源

MCU 的复位通常不止 POR / BOR，还可能有：
- 外部 RESET 引脚；
- 看门狗复位；
- 软件触发复位；
- 调试器复位；
- 时钟异常 / 错误复位；
- 低功耗唤醒相关特殊复位路径。

### 为什么区分复位源很重要
因为不同复位源意味着：
- 发生原因不同；
- 有些状态会被清得不一样；
- 启动后应采取的恢复逻辑可能不同。

例如：
- BOR 可能提示电源不稳；
- 看门狗复位说明程序可能卡死；
- 软件复位可能是升级 / 模式切换的正常流程。

所以很多 MCU 都会提供复位原因寄存器让你在启动后读取。

---

## 7. 电源稳定后，CPU 是怎么开始执行的

## 7.1 复位释放后，CPU 会去固定入口取向量表
以 Cortex-M 为例，启动时会从向量表读取：
- 第一个字：初始 MSP；
- 第二个字：Reset Handler 入口地址。

这意味着 MCU 真正的软件入口并不是 `main()`，而是：
- **Reset Handler**
- 也就是启动文件中的复位处理函数。

### 7.2 为什么先取栈顶
因为后面任何函数调用、局部变量、中断压栈都需要有效栈环境。
所以 CPU 一开始就要知道自己的主栈从哪里开始。

---

## 8. 启动代码在做什么

一个典型启动流程会做这些事：
1. 设置栈指针；
2. 搬运 `.data` 初值到 SRAM；
3. 清零 `.bss`；
4. 可能初始化 C 运行时环境；
5. 调用系统时钟初始化；
6. 进入 `main()`。

### 为什么这些不能省
如果不搬 `.data`：
- 你的带初值全局变量会是错的。

如果不清 `.bss`：
- 默认应为 0 的全局变量会变成垃圾值。

如果时钟没初始化到预期：
- CPU 频率、外设波特率、定时器节拍都会错。

所以启动代码不是“模板样板”，而是系统能否正常运行的根基。

## 9. 默认时钟和后续时钟切换

## 9.1 为什么上电后常先用内部时钟
很多 MCU 上电后会先启用：
- 内部 RC 振荡器；
- 一个较保守、能快速起振的默认时钟源。

原因很简单：
- 启动快；
- 不依赖外部晶振先稳定；
- 能先把系统带起来。

### 9.2 为什么后面常切到 HSE / PLL
当系统稳定后，可能会切到：
- 更高精度的外部晶振；
- 经 PLL 倍频后的高性能主频；

这是为了：
- 提升 CPU 性能；
- 满足 USB / 通信精度要求；
- 提供更准确的系统节拍。

### 9.3 为什么时钟切换很危险
如果切换过程没处理好，可能出现：
- 时钟源未就绪就切换；
- Flash 等待周期不够；
- 总线分频超限；
- 外设波特率全错。

所以时钟切换虽然常见，但绝不是无风险动作。

---

## 10. Flash wait state 为什么和启动有关

当 CPU 主频升高时，Flash 不一定能无等待地跟上取指速度。
所以系统通常要配置：
- Flash 等待周期；
- 预取 / cache；
- 某些访问加速机制。

如果频率提上去了，但 Flash wait state 没跟上，可能会：
- 程序异常；
- 跑飞；
- 难以解释的故障。

这也是为什么“时钟配置”和“存储访问配置”在启动阶段经常绑在一起考虑。

---

## 11. 从复位到 main() 的一条完整链

你可以把整条链记成：

> 电源上升  
> → POR 保持复位直到电压可用  
> → 复位释放  
> → CPU 取向量表  
> → Reset Handler 运行  
> → 初始化 `.data` / `.bss` / 运行时环境  
> → 配系统时钟  
> → 进入 `main()`  
> → 后续外设初始化与业务执行

而运行中如果电压跌落，则：

> 电压跌破 BOR 阈值  
> → 系统被强制拉回复位 / 重启路径

只要这条链清楚，很多“为什么会这样复位、为什么 main 前就已经做了事”的问题都会变得顺。

## 12. 为什么有些现象只在冷启动、热复位或掉压时出现

这是实际工程里很关键的一点。

### 12.1 冷启动
- 电源从 0 开始爬升；
- POR 路径完整参与；
- 外部器件、电源域、晶振都从头启动。

### 12.2 热复位 / 软件复位
- 电源可能并没掉；
- 外围器件可能还带状态；
- 某些寄存器 / 外设行为与冷启动不同。

### 12.3 掉压 / BOR
- 系统可能在运行中突然被打断；
- 某些操作（尤其 Flash 写）风险更高；
- 复位原因诊断价值很大。

所以同样是“重启”，底层背景并不一样，这就是很多诡异问题只在某一种启动路径出现的原因。

---

## 13. 一些非常典型的工程坑点

- 只理解 `main()`，不理解 Reset Handler。
- 不读取复位原因寄存器，导致排障时分不清是 BOR 还是看门狗。
- 电源设计太弱，运行中偶发 BOR。
- 时钟切换过早或顺序错误。
- 提高主频后忘记调 Flash wait state。
- 以为软件复位和冷启动效果完全一样。
- 低功耗唤醒后沿用错误的时钟假设。

---

## 14. 一个很实用的排查顺序

当你遇到“上电异常、重启异常、偶发复位”时，可以按这个顺序查：
1. 复位原因寄存器是什么。
2. 电源波形是否稳定，是否有掉压。
3. POR / BOR 阈值和供电设计是否匹配。
4. 默认时钟和切换时钟是否按预期执行。
5. Flash wait state 是否匹配当前主频。
6. `main()` 前启动代码是否正确完成 `.data` / `.bss` 初始化。
7. 冷启动和热复位路径是否存在外设残留差异。

这个顺序能把问题快速分到：
- 电源；
- 复位；
- 启动代码；
- 时钟配置；

四大类里。

# DMA、Cache 与一致性问题

## 1. 这篇文档解决什么问题

本专题回答的是：

> **为什么明明 CPU 写了数据，DMA 却读到旧值；或者 DMA 明明写完了，CPU 却还在看老数据？**

这正是很多中高级嵌入式和 Linux 平台会踩到的典型“深水区”问题。

它重点解决这些问题：
- DMA 到底是什么，它和 CPU 拷贝本质区别在哪。
- Cache 为什么会让“内存里的值”和“CPU 看到的值”不一致。
- 什么叫 Cache Coherency（一致性）。
- 为什么 MPU、Non-Cacheable 区、clean / invalidate 会成为必要操作。

---

## 2. 核心结论先看

### 2.1 DMA 的本质是“旁路 CPU 的搬运工”
DMA 负责在外设和内存、内存和内存之间搬数据，而不需要 CPU 一条条执行 load/store。

这意味着：
- CPU 眼里的数据路径；
- DMA 眼里的数据路径；

不一定始终经过同一层缓存体系。

### 2.2 Cache 提高了 CPU 速度，但也制造了“视图不一致”
CPU 访问某块内存时，可能先命中 Cache，而不是直接去 SRAM / DDR。

所以会出现两种情况：
- CPU 改了数据，但新值还停留在 Cache，没写回主存；
- DMA 把新数据写到了主存，但 CPU 还在读旧 Cache 行。

### 2.3 一致性问题的本质不是“谁错了”，而是“谁看的是哪一份副本”
- DMA 通常看主存；
- CPU 可能看 Cache；
- 如果没有同步动作，两边就可能不在看同一份最新数据。

---

## 3. DMA 到底是什么

## 3.1 为什么需要 DMA
如果每次外设收发都让 CPU 手工搬运数据，就会导致：
- CPU 被大量拷贝操作占满；
- 吞吐下降；
- 中断频率过高；
- 实时性变差。

DMA 的意义是：

> **把“搬运”这个机械动作交给专门硬件完成。**

### 3.2 DMA 常见路径
典型路径包括：
- 外设寄存器 → SRAM；
- SRAM → 外设寄存器；
- SRAM → SRAM；
- 某些平台还支持更复杂的 scatter-gather。

### 3.3 DMA 与中断的配合
DMA 自己负责搬运，但通常仍要通过中断告诉 CPU：
- 传输完成；
- 传输半完成；
- 发生错误。

所以驱动里经常会看到：
- MMIO 配寄存器；
- DMA 配通道；
- 中断收尾处理。

---

## 4. Cache 为什么会带来问题

## 4.1 Cache 的出发点
Cache 是为了让 CPU 更快：
- 避免每次都去慢速主存取数；
- 利用空间局部性和时间局部性；
- 把近期常用的数据和指令留在更近的存储层。

### 4.2 写回与读命中会制造“延迟同步”
在很多系统里，CPU 写内存时并不一定立刻把新值写到主存，而可能：
- 先只改 Cache 行；
- 标记该行为 dirty；
- 以后再择机回写。

这就是问题起点。

### 4.3 CPU 和 DMA 不一定共享同一个可见视图
如果 DMA 直接从主存读，而 CPU 的最新写入还在 Cache，那么：
- DMA 看到的就是旧值。

反之，如果 DMA 把新数据写回主存，而 CPU 还在读自己的旧 Cache 行，那么：
- CPU 看到的就是旧值。

---

## 5. 两类最典型的一致性故障

## 5.1 CPU → DMA 方向
场景：
- CPU 准备好一块发送缓冲区；
- DMA 准备拿这块缓冲发给外设。

风险：
- CPU 只把数据写进了 Cache；
- DMA 从主存拿到的还是旧数据。

结果：
- 外设发出去的是脏的 / 老的包；
- 逻辑看起来“偶发错包”。

### 解决思路
在启动 DMA 之前，通常需要：
- clean cache；
- 把 dirty 数据真正写回主存。

## 5.2 DMA → CPU 方向
场景：
- DMA 从外设收到数据，写进接收缓冲区；
- CPU 随后去读该缓冲。

风险：
- CPU 还保留着旧 Cache 行；
- 它读到的不是 DMA 刚写的新数据。

结果：
- 明明 DMA 已完成，CPU 却像没收到一样；
- 或收到的数据莫名其妙是旧内容。

### 解决思路
在 CPU 读取之前，通常需要：
- invalidate cache；
- 丢掉旧缓存行，从主存重新取。

---

## 6. clean、invalidate 到底是什么意思

### 6.1 clean
把 Cache 中脏数据写回主存，但缓存行本身可继续保留。

适合场景：
- CPU 改完数据，准备让 DMA 去读。

### 6.2 invalidate
丢弃 Cache 中对应缓存行，让 CPU 下次重新从主存取。

适合场景：
- DMA 刚写完主存，CPU 准备读。

### 6.3 为什么顺序很重要
如果顺序弄反，或者忘记做，就会出现那类“偶现、难复现、改打印就变好”的幽灵 bug。

这类问题之所以难查，是因为：
- 从代码逻辑上看一切都对；
- 错的是“可见性时机”，而不是算法本身。

---

## 7. MPU 和 Non-Cacheable 区为什么重要

## 7.1 最直接的方法：把 DMA 缓冲区设成不可缓存
如果某块内存专门给 DMA 用，常见做法是：
- 把它放在专门 section；
- 用 MPU 配成 Non-Cacheable；
- 避免 CPU 缓存它。

这样做的好处是：
- 简化一致性管理；
- 避免频繁 clean / invalidate；
- 更适合高频收发缓冲区。

### 7.2 为什么不是所有内存都设成不可缓存
因为 Cache 是性能来源。
如果把大块内存都设成 Non-Cacheable：
- CPU 访问会显著变慢；
- 系统整体性能下降。

所以真实工程里需要平衡：
- 哪些内存追求速度；
- 哪些内存追求 DMA 可见性稳定。

---

## 8. 为什么 `volatile` 解决不了这个问题

这是一个非常常见误区。

`volatile` 只能告诉编译器：
- 不要把这次读写优化掉；
- 每次都按源代码要求发起访问。

但它**并不能控制 CPU Cache 行为**，也不能告诉 DMA 去看哪一层副本。

所以：
- `volatile` 解决的是编译器可见性；
- Cache Coherency 解决的是硬件存储层次可见性。

这两者不是一个层面的问题。

---

## 9. 典型工程策略

## 9.1 小而频繁的 DMA 缓冲
常用策略：
- 单独 section；
- MPU 配 Non-Cacheable；
- 减少手工维护成本。

## 9.2 大块、偶发 DMA 传输
常用策略：
- 保持缓存；
- 在关键时机手工 clean / invalidate；
- 用驱动层 API 封装同步动作。

## 9.3 Linux / 大系统平台
通常还会借助：
- 专门 DMA API；
- 一致性内存分配接口；
- IOMMU / 总线属性配置；
- 缓冲区映射与解除映射流程。

其本质仍然是在处理同一个问题：

> **让 CPU 和 DMA 对同一块数据达成“看到的是同一份最新版本”。**

---

## 10. 如何识别这类问题

如果你看到这些现象，要高度怀疑一致性问题：
- 明明内存打印正确，但外设发出去内容不对；
- DMA 完成中断到了，但 CPU 读到的接收内容像旧数据；
- 加日志、改单步、降优化后问题反而消失；
- 只在带 Cache 的平台出现，不带 Cache 的 MCU 一切正常；
- 问题偶现且和数据大小、对齐、地址位置有关。

这类特征非常典型。

---

## 11. 本专题最容易踩坑的点
- 以为 DMA 和 CPU 天然总能看到同一份数据。
- 把 `volatile` 当成一致性问题的解决方案。
- 分不清 clean 和 invalidate 的方向和时机。
- 没给 DMA 缓冲做独立 section / 属性规划。
- 在带 Cache 平台上照搬无 Cache MCU 的经验。
- 忽视对齐、Cache line 粒度和缓冲区边界问题。

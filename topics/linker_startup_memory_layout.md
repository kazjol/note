# 链接脚本、启动流程与内存布局

## 1. 这篇文档解决什么问题

本专题回答的是：

> **程序编译完成以后，为什么还不能直接“凭空运行”，它是怎么被放到正确内存位置并真正跑起来的？**

它同时解释这些关键问题：
- `.text`、`.data`、`.bss`、Heap、Stack 各自是什么。
- 为什么 `.data` 会同时占用 Flash 和 SRAM。
- 什么是 LMA / VMA。
- `Startup.s` 在 `main()` 之前到底做了什么。
- 为什么 `section`、链接脚本、`.map` 文件决定了程序是否真的按预期装进内存。

---

## 2. 核心结论先看

### 2.1 程序运行不只依赖堆和栈
程序至少包含这些区域：
- **`.text`**：代码指令，通常放在 Flash。
- **`.rodata`**：只读常量，通常也在 Flash。
- **`.data`**：有初值的全局 / 静态变量，运行时在 SRAM。
- **`.bss`**：无初值或初值为 0 的全局 / 静态变量，运行时在 SRAM。
- **Heap**：动态分配区。
- **Stack**：函数调用链和局部变量常用区域。

所以“不是堆就是栈”是典型误区。

### 2.2 编译结果和运行状态不是同一回事
编译 / 链接阶段决定的是：
- 每个符号该落在哪个段；
- 每个段该放在哪个地址；
- 镜像文件里数据如何排列。

运行阶段则还需要：
- 上电后把初值从 Flash 搬到 SRAM；
- 把 `.bss` 清零；
- 设置栈；
- 进入 `main()`。

### 2.3 `Startup.s` 是程序真正的“起跑线”
很多人以为程序从 `main()` 开始，其实不是。
真正的最早入口通常是复位向量指向的启动代码。它负责：
- 设定初始栈顶；
- 搬运 `.data`；
- 清零 `.bss`；
- 可能还要初始化时钟、C 运行时环境；
- 最后再跳进 `main()`。

---

## 3. 各内存段到底是什么

## 3.1 `.text`
- 存放机器指令。
- 通常位于 Flash。
- 一般不会在运行时被修改。

### 关键理解
`static` 不能把代码从 `.text` 变到 `.bss`。`static` 影响的是作用域 / 生命周期，不会改变“代码本身就是代码段”这一事实。

## 3.2 `.data`
- 存放**有初始值**的全局变量和静态变量。
- 运行时必须在 SRAM 里，因为它们可能会被修改。

### 为什么 `.data` 会同时占用 Flash 和 SRAM
因为它有两份含义：
- Flash 里保存“初始值模板”；
- 上电后被复制到 SRAM，供程序运行时读写。

这就是你问到的“双重占用”的本质。

## 3.3 `.bss`
- 存放无初值或初值为 0 的全局 / 静态变量。
- 运行时在 SRAM。
- 镜像文件里不需要真的存一大片 0，只要启动时清零即可。

这也是为什么 `.bss` 常常不增加镜像文件体积，但会真实占用 SRAM。

## 3.4 Heap 与 Stack
**Heap**：
- 动态分配区；
- 通过 `malloc` / `new` 等使用；
- 生命周期由程序控制。

**Stack**：
- 函数调用、返回地址、局部变量、保存寄存器的区域；
- 生命周期随调用链变化。

---

## 4. LMA 与 VMA：双户口本模型

### 4.1 什么是 LMA
**LMA（Load Memory Address）** 是镜像被存放的位置，通常在 Flash。

### 4.2 什么是 VMA
**VMA（Virtual / Run Memory Address）** 是程序实际运行时访问的位置。在 MCU 场景里，通常指运行地址。

### 4.3 为什么会不同
例如 `.data`：
- 初值模板存放在 Flash；
- 运行时要在 SRAM。

所以同一段内容会有：
- 一份“装载地址”；
- 一份“运行地址”。

这就是 LMA / VMA 分离的根本原因。

### 4.4 RAM 函数也是同理
有些对时延敏感、不能在 Flash 执行的函数会被放到 RAM 中执行。
它们也常常是：
- 镜像先存于 Flash；
- 启动时拷贝到 SRAM；
- 随后从 SRAM 执行。

## 5. `Startup.s` 在 `main()` 前做什么

一个典型启动过程如下：
1. CPU 复位；
2. 从向量表中取出初始 MSP 和复位处理函数地址；
3. 设置栈顶；
4. 把 `.data` 从 Flash 复制到 SRAM；
5. 把 `.bss` 清零；
6. 可能初始化时钟和底层运行时；
7. 调用 `main()`。

### 为什么这一步绝对不能省
如果不搬 `.data`：
- 你的全局初值全错。

如果不清 `.bss`：
- 本应为 0 的全局变量会带着随机垃圾值。

所以这不是“优化动作”，而是程序正确运行的前提。

---

## 6. 链接脚本在决定什么

链接脚本负责回答这些问题：
- Flash 从哪里开始，SRAM 从哪里开始。
- 哪些段放 Flash，哪些放 SRAM。
- 启动代码、向量表、普通代码、只读数据、读写数据分别放哪。
- 哪些段需要装载后拷贝。
- 堆和栈预留多少空间。

换句话说，链接脚本就是：

> **程序最终物理布局的总设计图。**

没有它，编译器生成的各个目标文件无法被正确拼装成一个能运行的系统镜像。

---

## 7. 为什么要用 `section` 做细粒度分段

你问到：为什么不全部放默认段。
答案是因为真实工程需要：
- 把特定函数放入 RAM 加速执行；
- 把某些数据放入特定 SRAM / TCM / 共享内存区域；
- 把任务栈隔离放置，方便 MPU 做越界保护；
- 把 DMA 缓冲区放在 Non-Cacheable 区域；
- 对异构多核共享区做明确约束。

所以细粒度分段不是“炫技”，而是为了：
- 性能；
- 安全；
- 可控；
- 与硬件拓扑匹配。

## 8. 编译与链接优化如何真正生效

### 8.1 `-ffunction-sections` / `-fdata-sections`
这两个选项会让：
- 每个函数放到独立 section；
- 每个数据对象尽量放到独立 section。

这样链接器才更容易把没被引用的内容裁掉。

### 8.2 `-Wl,--gc-sections`
这是告诉 GCC 前端：
- 把 `--gc-sections` 透传给底层链接器；
- 让链接器回收无用 section。

如果少了 `-Wl,`，参数可能根本没传进去。

### 8.3 `.map` 文件为什么重要
`.map` 文件是最关键的验证工具之一。你可以用它确认：
- 每个符号 / section 最终落在哪；
- 哪些段占用了多少 Flash / SRAM；
- `Discarded input sections` 里是否真的出现了被裁掉的废代码；
- 优化是否只是“以为生效”，还是真正生效。

## 9. `.hex` 与 `.bin` 为什么差这么多

### `.bin`
- 纯二进制内容；
- 紧凑；
- 不携带显式地址信息；
- 更适合烧录工具直接按已知基地址写入。

### `.hex`
- 文本格式；
- 每行有地址、长度、校验；
- 可表达地址空洞；
- 因为文本编码与记录开销，体积往往明显更大。

所以 `.hex` 大，并不代表里面的机器码真的更多。

## 10. DMA 与 Cache 一致性为什么会踩坑

在带 Cache 的系统里：
- CPU 写数据时，可能只写进 Cache，还没回写到 SRAM；
- DMA 却直接去 SRAM 读；
- 结果 DMA 看到的是旧数据。

反过来也可能出现：
- DMA 写完 SRAM；
- CPU 还在读旧 Cache 行。

常见解决方法：
- Cache clean / invalidate；
- 把 DMA 区放到 Non-Cacheable 内存；
- 用 MPU 对某些区域做属性配置。

---

## 11. 本专题最容易踩坑的点
- 把程序运行空间简化成“只有堆和栈”。
- 不理解 `.data` 的 Flash + SRAM 双重属性。
- 不知道 `main()` 前其实还有启动代码。
- 把 LMA / VMA 混为一谈。
- 用了 `section` 但没在链接脚本里接住。
- 以为加了优化选项就一定生效，却不看 `.map` 文件验证。
- 在带 Cache 的平台上忽视 DMA 一致性问题。

# Linux 内存管理与页表

## 1. 这篇文档解决什么问题

本专题回答的是：

> **程序看到的虚拟地址到底是什么，它和真实物理内存有什么关系，内核又是如何用页表和权限把整个系统组织起来的？**

它重点解释这些问题：
- 为什么用户程序看到的是虚拟地址而不是物理地址。
- 为什么虚拟地址连续不代表物理地址连续。
- 页表、缺页中断、换页、内核空间映射分别在做什么。
- `malloc`、`calloc`、`mmap`、`kzalloc`、DMA 连续内存在语义上为什么不同。

---

## 2. 核心结论先看

### 2.1 程序平时接触的几乎都是虚拟地址
无论是用户态指针还是内核中的大部分普通地址，本质上都不是“直接物理地址”，而是：
- 先由 CPU 发出虚拟地址；
- 再由 MMU 查页表；
- 转换成物理地址后访问内存。

### 2.2 虚拟内存的价值不只是“空间变大”
它更重要的价值是：
- 隔离进程；
- 保护内核；
- 提供按需分配；
- 让离散物理页拼成连续虚拟区间；
- 支持共享映射、文件映射、匿名映射等机制。

### 2.3 页表本质上是一张“地址翻译 + 权限规则表”
页表不仅决定：
- 这段虚拟地址映射到哪段物理页；

还决定：
- 能不能读；
- 能不能写；
- 能不能执行；
- 用户态能不能访问；
- 是否可缓存、是否可共享等。

---

## 3. 为什么要有虚拟地址

## 3.1 如果没有虚拟地址会怎样
如果程序直接操作物理地址，会出现很多问题：
- 不同进程容易互相覆盖内存；
- 用户程序能直接碰内核和别的进程；
- 程序装载地址难以灵活安排；
- 无法优雅实现共享库、按需加载和保护。

所以虚拟地址的核心作用是：

> **给每个进程一个“看起来独立、连续、受保护”的地址世界。**

### 3.2 同一个虚拟地址，在不同进程里可以指向不同物理页
这正是进程隔离的关键。比如两个进程都看到自己的某块堆地址是类似的虚拟区间，但底层可能映射到完全不同的物理页。

---

## 4. 页（Page）与页表（Page Table）到底是什么

## 4.1 为什么按“页”管理而不是按字节管理
如果地址转换和权限控制按字节做，开销会极其巨大。
所以系统按固定粒度管理内存，常见粒度是 4KB 页。

好处是：
- 管理成本可控；
- 映射单位统一；
- 便于按页换入换出；
- 适合缓存和 TLB 协同。

## 4.2 页表的本质
页表就是一层或多层的数据结构，用来记录：
- 某段虚拟页号对应哪段物理页号；
- 这页的权限和属性。

当 CPU 访问一个虚拟地址时，会先把虚拟地址拆成：
- 页号；
- 页内偏移。

MMU 用页号查页表，得到物理页基址，再加上页内偏移形成最终物理地址。

---

## 5. 多级页表为什么存在

如果整个虚拟地址空间都用一张巨大平面表来映射，内存开销会很夸张。
所以现代系统通常使用多级页表：
- 顶层目录；
- 中间层目录；
- 叶子页表项。

这样只有真正用到的地址区间才需要分配对应页表层级，节省内存。

这也是为什么页表本身也会占用内存，而且系统的地址管理并不是“零成本”的。

---

## 6. TLB：为什么地址转换不会慢得离谱

如果每次访存都完整走一遍多级页表，性能会很差。
所以 CPU 一般还有 TLB（Translation Lookaside Buffer）这种地址翻译缓存。

它缓存的是：
- 最近用过的虚拟页 → 物理页翻译结果。

所以整体路径变成：
1. 先查 TLB；
2. 命中则直接得到物理页；
3. 未命中再去查页表；
4. 再把结果回填到 TLB。

这说明“缓存”不只存在于数据层，也存在于地址翻译层。

---

## 7. 缺页中断到底在干什么

## 7.1 缺页不是一定出错
很多人一听“缺页”就觉得是异常错误。其实不是。

缺页中断的含义是：
- 当前访问的虚拟页没有现成可用翻译或未在内存中；
- CPU 触发异常，让内核介入处理。

### 7.2 可能出现的几种情况
1. **合法但尚未分配**：例如匿名页首次访问，内核现分物理页并建立映射。
2. **合法但在磁盘 / swap**：内核把它换回内存。
3. **权限不允许**：例如写只读页，触发保护异常。
4. **地址非法**：例如野指针，通常导致段错误。

所以缺页中断既可能是正常懒分配机制的一部分，也可能是真正程序错误的表现。

---

## 8. 为什么虚拟连续不代表物理连续

这点非常关键。

例如一个 64KB 的虚拟缓冲区，对程序来说是一大片连续地址；但底层可能只是：
- 16 个分散的 4KB 物理页；
- 由页表把它们拼起来。

这对普通 CPU 访问通常没问题，但对某些硬件场景就会有影响，比如：
- DMA 可能要求物理连续；
- 某些设备不能理解页表；
- IOMMU / 映射 API 必须介入。

所以“连续”这个词一定要分清：
- 虚拟连续；
- 物理连续；
- 总线可见连续。

---

## 9. 用户空间与内核空间为什么能隔离

### 9.1 不是因为“约定俗成”，而是 MMU 强制执行
页表可以给每页打权限标记，例如：
- 用户可读写；
- 仅内核可访问；
- 只读；
- 不可执行。

因此用户程序即便“知道”某个内核地址，也不能随便访问，因为权限检查会在硬件层面拦住它。

### 9.2 内核空间为什么经常在所有进程中都存在映射
很多体系结构里，内核空间会在每个进程页表中保持一套稳定映射，但标记成用户态不可访问。这样做可以减少内核态切换时的映射成本。

它的关键不是“是否映射”，而是“当前权限级是否允许访问”。

---

## 10. 用户态常见分配方式

## 10.1 `malloc`
- 从堆拿一块用户虚拟内存；
- 默认不清零；
- 可能来自 brk 扩展，也可能来自 mmap。

## 10.2 `calloc`
- 分配并清零；
- 常用来避免未初始化数据。

## 10.3 `mmap`
- 建立一段映射；
- 可匿名，也可映射文件；
- 是虚拟内存机制的重要接口。

这意味着用户态分配并不总是“向系统要一块物理连续内存”，而通常是在建立虚拟地址区间，并由内核按页管理底层物理资源。

---

## 11. 内核里的 `kzalloc`、vmalloc、连续内存

### 11.1 `kzalloc`
通常表示：
- 分配内核内存；
- 清零；
- 对较小对象常较适合。

从理解上你可以把它看成：
- 更偏对象级内存分配；
- 对硬件交互友好；
- 语义上强调安全初始化。

### 11.2 `vmalloc`
更偏向：
- 给内核提供虚拟连续区域；
- 底层物理页不一定连续。

### 11.3 为什么有时必须要物理连续
因为某些 DMA 控制器或设备：
- 不会自己查 CPU 页表；
- 只认识物理总线地址；
- 无法自然处理离散页。

这时就需要：
- 连续物理内存；
- 或 DMA API / IOMMU 提供设备可见映射。

---

## 12. 文件映射与匿名映射

这是虚拟内存很优雅的一点。

### 12.1 文件映射
虚拟页可以直接映射到文件内容：
- 访问时按页加载；
- 写回策略受映射方式影响；
- 很适合共享库、内存映射文件。

### 12.2 匿名映射
不对应具体文件，常用于：
- 堆；
- 大块临时缓冲；
- 进程私有数据。

这说明虚拟内存不只是“RAM 管理”，而是一种统一的地址资源抽象。

---

## 13. Copy-on-Write（写时复制）为什么高效

进程创建时，系统不一定立刻把所有内存真的复制一份，而可能：
- 先共享同一批物理页；
- 标成只读；
- 谁先写，谁再触发异常并复制出自己的页。

这就是写时复制。

它的好处是：
- `fork()` 成本下降；
- 很多只读或短生命周期分支无需真的复制全部内容。

本质上还是在利用：
- 页表；
- 权限位；
- 缺页异常。

---

## 14. 页缓存、Swap 与性能直觉

### 14.1 页缓存
文件数据可能先留在页缓存里，提升后续访问速度。

### 14.2 Swap
当物理内存紧张时，一部分不活跃页可能被换到磁盘交换区。

### 14.3 为什么 Swap 慢
因为磁盘和 RAM 的性能差距极大。出现频繁换入换出时，系统会非常卡。

所以缺页和换页机制本身是必要的，但它们绝不是“免费午餐”。

---

## 15. 这套机制和嵌入式思维怎么接起来

你之前问的很多问题都来自 MCU / RTOS 世界，那 Linux 的页表机制可以这样和前面的知识串起来：
- MCU 更强调固定物理地址和显式段布局；
- Linux 更强调虚拟地址、按页管理、权限隔离；
- 但两者都在解决“代码如何安全、可控、有效地使用内存”。

只是 Linux 把这件事做得更动态、更抽象、更通用。

---

## 16. 本专题最容易踩坑的点
- 把虚拟地址误当成物理地址。
- 以为连续指针区间一定对应连续物理内存。
- 把缺页中断都当成程序出错。
- 不理解页表既负责翻译也负责权限控制。
- 忽略 DMA 与页式内存管理之间的张力。
- 混淆 `kzalloc`、`vmalloc`、用户态 `malloc` 的语义边界。

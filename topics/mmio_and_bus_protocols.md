# MMIO 与总线协议

## 1. 这篇文档解决什么问题

本专题回答一个最核心的问题：

> **一段 C 代码到底是如何一步步落到真实硬件上的？**

它同时解释这些容易混淆的点：
- MMIO 和 I2C / SPI / CAN / UART 到底谁更底层。
- 改某个地址的某一位，和“发一个协议包”本质区别是什么。
- GPIO、寄存器、总线、DMA、共享内存之间是什么关系。

---

## 2. 核心结论先看

### 2.1 MMIO 比协议通信更底层
- **MMIO（Memory-Mapped I/O）**：CPU 直接把某段地址空间当成外设寄存器来访问。
- **协议通信（I2C / SPI / UART / CAN）**：CPU 通常先配置本地控制器寄存器，再由硬件控制器按协议时序把数据发到线缆上。

所以从层级上看：

> CPU 指令 → MMIO 寄存器访问 → 本地外设控制器动作 → 总线协议时序 → 对端设备响应

也就是说，很多“协议通信”本质上仍然是以 MMIO 为起点。

### 2.2 地址访问与协议访问不是一回事
- **地址访问**：你改的是本地 CPU 可见地址空间里的寄存器 / SRAM / 外设窗口。
- **协议访问**：你并没有直接碰到对方芯片内部物理地址，而是通过线上协议把命令发过去，由对方芯片内部逻辑自行解析。

### 2.3 共享内存通信本质是“约定 + 轮询 / 中断”
多核或异构核之间通过共享 SRAM 通信时，真正生效的不是“改了位就自动完成逻辑”，而是：
- 双方先约定好某块共享内存的格式；
- 一方写入标志位 / 命令字 / 数据区；
- 另一方通过轮询、中断或硬件信号量看到变化后执行对应动作。

---

## 3. MMIO 的本质

### 3.1 什么是 MMIO
MMIO 的核心思想是：

> **把外设寄存器映射进 CPU 的地址空间。**

这样 CPU 执行一条普通的 load/store 指令，就可以完成对硬件寄存器的读写。

例如：
```c
#define GPIOA_ODR (*(volatile unsigned int *)0x48000014)
GPIOA_ODR |= (1U << 5);
```

从软件视角看只是改变量；从硬件视角看则是：
- CPU 发起一次总线读操作；
- 取回寄存器当前值；
- 在寄存器值上做按位或；
- 再发起一次总线写操作写回外设。

### 3.2 为什么必须加 `volatile`
`volatile` 不是为了“线程安全”，而是为了告诉编译器：
- 这个地址对应的内容可能随时被硬件改变；
- 不能把读写优化掉；
- 每次访问都必须真的发起内存 / 总线访问。

### 3.3 结构体映射的意义
很多底层代码会这样写：
```c
typedef struct {
    volatile unsigned int CR;
    volatile unsigned int SR;
    volatile unsigned int DR;
} UART_Reg;

#define UART0 ((UART_Reg *)0x40001000)
UART0->DR = 0x55;
```

这样做的本质是：
- 用结构体字段名提升可读性；
- 用固定偏移对应硬件寄存器布局；
- 让“地址 + 偏移”变成“字段访问”。

## 4. SRAM 位操作、共享内存与跨核通信

### 4.1 改 SRAM 某一位是不是 MMIO
严格说：
- **普通 SRAM 地址** 不是外设寄存器，不属于 MMIO。
- 但它仍然是“按地址访问内存”。

如果某块 SRAM 被多个核共享，并且双方把它当成“命令邮箱”，那么软件语义上它就类似“共享控制区”，但物理上它依然是存储器，而不是外设寄存器。

### 4.2 为什么改一个位就能触发功能
本质原因不是“这个位有魔法”，而是：
- 系统设计者先约定了该位的语义；
- 另一侧程序持续盯着这个位；
- 一旦发现位变化，就进入相应逻辑。

这就是**共享内存 + 协议约定**的本质。

### 4.3 并发风险：RMW 不是原子操作
像下面这种代码：
```c
shared_flag |= (1U << 3);
```
底层通常不是一步完成，而是：
1. 读旧值；
2. 修改某一位；
3. 写回新值。

如果两个执行流同时做类似操作，就可能互相覆盖。常见解决方法：
- 关中断；
- 自旋锁 / 互斥锁；
- 原子操作；
- 硬件信号量；
- 位带或专用 set/clear 寄存器。

## 5. I2C、SPI、UART、CAN 的本质区别

## 5.1 I2C
- 两根线：SCL、SDA。
- 开漏输出，依赖上拉电阻。
- 支持多从机寻址。
- 总线空闲时通常两根线都为高。

### 关键理解
I2C 不是“直接改对方内存”，而是：
- 主机通过本地 I2C 控制器发起起始条件、地址、数据、停止条件；
- 从机收到后，在自己内部状态机里解析；
- 必要时再把数据写到自己的寄存器、FIFO 或 SRAM。

### 常见坑点
- SDA 被持续拉低导致总线挂死；
- 主机异常复位导致从机状态机残留；
- 需要手动发 9 个 SCL 脉冲做 Bus Clear；
- I2C + DMA 可能让数据“悄无声息”进入内存，调试时容易误判。

## 5.2 SPI
- 常见四线：CS、SCK、MISO、MOSI。
- 全双工。
- 没有 I2C 那样的设备地址，通常靠 CS 选通。

### 关键理解
SPI 的底层是移位寄存器交换：
- 主机每打一拍时钟，主从双方都移出 1 bit、移入 1 bit；
- 所以**想收就必须发**，即使只是读取，也常常要发 Dummy Byte。

## 5.3 UART
- 常见两线：TX、RX。
- 异步串行，无时钟线。
- 靠约定好的波特率、起始位、停止位工作。

### 关键理解
UART 很适合日志输出、调试串口、点对点模块通信，但不适合天然多从机挂接。

## 5.4 CAN
- 差分通信，抗干扰强。
- 广播式，带仲裁机制。
- 数据不是发给“某根片选线”，而是挂在总线上的所有节点都能看到，再由硬件过滤器决定谁收。

### 关键理解
CAN 不能让 CPU 把所有报文都吃下来，否则中断会非常频繁。实际工程里必须配置**硬件接收过滤器**减负。

## 6. GPIO、电平与板级问题

### 6.1 推挽输出与开漏输出
**推挽输出**：
- 内部同时具备上拉到 `VDD` 和下拉到 `VSS` 的驱动能力；
- 输出强、速度快；
- 不适合多器件直接线与。

**开漏输出**：
- 只能主动拉低；
- 输出高电平时其实是高阻；
- 必须依赖外部上拉；
- 适合 I2C 这种多设备共享总线。

### 6.2 I2C 线为什么会半高或卡低
常见原因包括：
- 上拉太弱 / 太强；
- 负载过大；
- 器件漏电；
- 从机时钟拉伸；
- 某个器件仍在主动拉低；
- 硬件短路或焊接异常。

你在实际调试里看到的“半高”，往往已经是电气层、驱动方式、时序状态机三者共同作用的结果。

## 7. DMA 在总线体系中的位置

DMA 的本质是：

> **把“搬内存”这件事交给专门硬件，而不是让 CPU 一条条指令去拷贝。**

所以很多通信链路实际上是：
- CPU 用 MMIO 配好 I2C / SPI / UART / DMA 控制器；
- 总线控制器负责收发；
- DMA 把数据从寄存器 FIFO 搬到 SRAM；
- 完成后再由中断告诉 CPU。

这就是为什么驱动里常常会看到“中断 + DMA + 寄存器配置”一起出现。

## 8. 一条完整的理解链

建议把本专题内容串成这条线：

> GPIO 电气特性  
> → 总线时序  
> → 本地外设控制器  
> → MMIO 寄存器访问  
> → DMA / 中断 / 轮询  
> → 上层驱动接口

只要这条线打通，你对很多“为什么这样写驱动”的问题就会自然通了。

---

## 9. 本专题最容易踩坑的点
- 把“协议访问”误解成“直接访问对方内存”。
- 把 SRAM 共享区误当成外设寄存器。
- 忽略 RMW 的非原子性。
- 忘记 `volatile` 的真实作用。
- 不了解 SPI 的“收必须伴随发”。
- 低估 I2C 总线挂死和电平异常的复杂性。
- 忽视 DMA 带来的“数据不是 CPU 写进去的”这一调试难点。

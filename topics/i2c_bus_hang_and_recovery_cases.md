# I2C 总线挂死与恢复案例

## 1. 这篇文档解决什么问题

本专题回答的是：

> **为什么 I2C 明明刚开始还能正常通信，跑一段时间后就突然挂死？挂死后 SCL / SDA 常见会呈现什么波形，根因通常有哪些，又该怎么恢复？**

这是板级调试里最典型、最折磨人的问题之一。

本篇重点不只是讲原理，还会按“现象 → 可能原因 → 恢复动作 → 预防设计”的方式整理。

---

## 2. 核心结论先看

### 2.1 I2C 挂死通常不是“单点软件 bug”，而是协议状态机、电气条件和异常中断共同作用的结果
同样一个“总线挂死”现象，背后可能来自：
- 主机中途复位；
- 从机状态机残留；
- SDA / SCL 被外部器件拉住；
- 上拉不合适；
- 时钟拉伸处理不当；
- DMA / 中断 / 低功耗切换把事务打断；
- 板级漏电、短路或焊接问题。

### 2.2 最经典的挂死图景是：SCL 高，SDA 长时间低
这往往表示：
- 主机以为事务结束了；
- 但某个从机还以为自己处在字节传输中；
- 它还在等待时钟继续，把一个“0”保持在 SDA 上。

这就是为什么“补 9 个 SCL 脉冲”经常有效。

### 2.3 I2C 恢复动作本质上是在“帮总线把残留事务走完，重新回到空闲态”
所以恢复不是魔法，而是：
- 让从机状态机吐完剩余 bit；
- 释放 SDA；
- 再重新产生 Stop；
- 让总线回到 SCL/SDA 都高的空闲状态。

---

## 3. 挂死前先记住“正常空闲状态”

I2C 正常空闲时应满足：
- SCL = 高
- SDA = 高

这是因为：
- 两根线通常都是开漏；
- 没人主动拉低时，由上拉电阻把线拉高。

所以任何长期偏离这个空闲状态的情况，都值得怀疑。

---

## 4. 常见挂死现象分类

## 4.1 现象 A：SCL 高，SDA 低

### 典型含义
最常见解释是：
- 从机还没完成当前字节 / ACK / 数据位流程；
- 正在持续拉低 SDA；
- 主机却已经退出、复位或放弃事务。

### 典型诱因
- 主机在传输中复位；
- 中断 / DMA 异常打断事务；
- Stop 条件没真正发出去；
- 时序错误导致从机没识别到结束。

### 常用恢复
- 临时接管 SCL 为 GPIO；
- 手动输出 9 个脉冲；
- 观察 SDA 是否释放；
- 然后再手动造一个 Stop。

---

## 4.2 现象 B：SCL 低，SDA 高或低

### 典型含义
这通常说明：
- 某设备在拉低 SCL；
- 可能是时钟拉伸；
- 也可能是硬件状态机卡死；
- 还可能是板级短路或强下拉。

### 先别急着判死刑
如果设备支持 clock stretching，那么短时间 SCL 低未必是故障。
真正要看的是：
- 它是不是长时间不释放；
- 是否超出协议 / 驱动允许等待时间；
- 对应从机是否确实有理由忙这么久。

---

## 4.3 现象 C：SCL / SDA 半高、边沿发软、波形难看

### 典型含义
这更偏电气问题，例如：
- 上拉太弱；
- 线容过大；
- 负载太重；
- 某器件漏电；
- 焊接虚接；
- 测量探头影响；
- 多器件共同拖拽这根线。

### 关键点
这类问题表面看像“通信失败”，但根因可能压根不在协议状态机，而在：
- 电平建立不起来；
- 采样边沿错过；
- 高电平不够高；
- 下降 / 上升时间不满足速率要求。

---

## 5. 最经典的挂死场景：主机中途复位

这是 I2C 挂死里最常见的一类。

### 场景过程
1. 主机正在给从机发某个字节；
2. 中间发生异常复位 / 看门狗复位 / 软件重启；
3. 主机不再继续发时钟，也没发 Stop；
4. 从机还停留在“我还在事务中”的状态；
5. 某个 bit，尤其是 0，被继续保持在 SDA 上；
6. 新起来的主机一看总线不空闲，于是后续通信全乱。

### 为什么补 9 个时钟有用
因为从机在等时钟把剩余 bit / ACK 流程走完。
补时钟的目的就是：
- 人工把这个未完成字节推到结束；
- 让从机状态机有机会回到空闲。

9 个脉冲是因为：
- 一个字节 8 bit；
- 再加一个 ACK 位周期。

它不是绝对宇宙真理，但在大多数寄存器型从机场景下很有效。

## 6. 恢复动作怎么做才更稳

## 6.1 通用恢复思路
1. 禁用当前 I2C 外设控制器；
2. 把 SCL / SDA 临时切到 GPIO 模式；
3. 确保 SDA 为输入 / 释放态；
4. 手动对 SCL 打若干脉冲；
5. 观察 SDA 是否释放为高；
6. 在 SCL 为高时造一个 Stop；
7. 重新初始化 I2C 控制器；
8. 再尝试重新发事务。

## 6.2 为什么 Stop 很关键
即使 SDA 释放了，如果没有一个明确结束语义，某些从机状态机仍可能不算真正回到空闲。

Stop 的意义是：
- 正式告诉总线“本次事务结束”。

---

## 7. 示例级伪代码思路

```c
// 1. 关闭 I2C 外设
// 2. SCL/SDA 切 GPIO
// 3. 释放 SDA
for (int i = 0; i < 9; i++) {
    drive_scl_low();
    short_delay();
    release_scl_high();
    short_delay();
    if (read_sda_high()) {
        // 从机已释放 SDA，可考虑提前结束
    }
}

// 4. 人工造 Stop
release_sda_low_if_needed();
release_scl_high();
short_delay();
release_sda_high();

// 5. 恢复 I2C 控制器
reinit_i2c();
```

### 注意
实际实现时还要结合：
- 引脚是开漏还是普通 GPIO；
- 上拉是否可靠；
- 时序延时是否足够；
- 平台寄存器切换是否安全。

---

## 8. 除了主机复位，还有哪些典型根因

## 8.1 从机真的忙，但驱动不理解 clock stretching
如果从机因为内部处理需要拉住 SCL，而主机驱动：
- 没正确支持 clock stretching；
- 或等待超时处理写得差；

就可能把“忙”误判成“挂死”。

## 8.2 DMA / 中断把状态机打穿
例如：
- DMA 还没结束，软件先改状态；
- 错误中断没有清；
- 事务超时处理和正常完成路径同时触发；
- ISR 和任务同时访问同一 I2C 控制块。

这种情况表面是总线异常，根因其实是驱动并发没收好。

## 8.3 低功耗切换把总线打断
例如：
- 事务还没结束就关时钟；
- 外设电源域先掉；
- 主核休眠时 I2C 控制器状态没保住；
- 唤醒后驱动以为总线是干净的。

### 这类问题最容易“偶现”
因为它和时序窗口强相关，不一定每次都能重现。

## 8.4 板级硬件问题
不要低估这些概率：
- 上拉阻值不对；
- 总线过长；
- 某器件焊反 / 虚焊；
- 电源不稳；
- 某器件掉电后通过 IO 反灌；
- 线被别的功能误复用。

---

## 9. 如何区分“状态机挂死”还是“电气挂死”

一个很实用的思路：

### 更像状态机问题时
- 补时钟后能恢复；
- 复位某个从机后恢复；
- 总线波形电平本身还算正常；
- 常发生在主机异常中断事务后。

### 更像电气问题时
- 波形长期半高、边沿很差；
- 换速率会明显变化；
- 换上拉电阻或短线后改善；
- 某块板好某块板坏；
- 单独断开某器件后总线恢复。

这个区分很重要，因为两类问题的处理方向完全不同。

---

## 10. 一个实用排查顺序

当你遇到 I2C 挂死，建议按这个顺序查：
1. 看空闲时 SCL / SDA 是否都能稳定为高。
2. 看挂死时是 SCL 低还是 SDA 低。
3. 确认主机是否中途复位 / 超时退出 / 关时钟。
4. 确认从机是否支持或正在做 clock stretching。
5. 检查上拉、电平、板级波形质量。
6. 检查 I2C 驱动是否存在 ISR / DMA / 任务并发问题。
7. 尝试 9 脉冲 + Stop 恢复，看是否可解。
8. 单独隔离可疑从机，看是哪颗器件拖住总线。

这个顺序通常能很快把范围缩小。

## 11. 预防设计建议

## 11.1 软件侧
- 给事务做超时；
- 超时后做 bus recovery；
- 明确处理中途复位后的恢复路径；
- 防止 I2C 控制块被多上下文乱改；
- 错误中断和正常完成路径做清晰分层。

## 11.2 硬件侧
- 合理选上拉；
- 控制走线长度和容性负载；
- 做好电源时序；
- 避免掉电器件通过 IO 反灌总线；
- 必要时给关键从机提供可控复位。

## 11.3 系统设计侧
- 低功耗切换前确保事务结束；
- 多核 / 多任务访问同一 I2C 资源要有统一仲裁；
- 对关键外设保留“复位从机”或“总线恢复”手段。

---

## 12. 一个好记的总结

你可以把 I2C 挂死理解成三大类：
- **状态机没走完**：补时钟、补 Stop。
- **驱动并发 / 时序把流程打断**：修软件路径。
- **电平本身就不健康**：查上拉、负载、板级硬件。

只要先把这三类分清，定位效率会高很多。

---

## 13. 最容易踩坑的点
- 一看到挂死就只改软件重试次数。
- 不区分 SCL 被拉低还是 SDA 被拉低。
- 把 clock stretching 误判成故障。
- 不做 bus recovery，导致一次异常后整机后续都废。
- 忽视主机复位 / 低功耗 / DMA 并发带来的中断事务问题。
- 不结合波形，只看寄存器状态瞎猜。

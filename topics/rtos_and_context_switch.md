# RTOS 与上下文切换

## 1. 这篇文档解决什么问题

本专题解释：

> **系统为什么看起来能“同时”做很多事，以及这种同时到底是怎么实现的。**

你关心的不只是“任务切换”这个名词，而是：
- CPU 到底在什么时候切换任务；
- 切换时保存了什么；
- 为什么 RTOS 的延时不是单纯死等；
- 互斥锁、自旋锁、原子操作、中断上下文之间是什么边界。

---

## 2. 核心结论先看

### 2.1 单核上的并发本质是“快速切换”
绝大多数 MCU / CPU 在某个时刻只真正执行一条指令流。
所谓“多个任务同时运行”，本质上通常是：
- 定时器周期性打断当前执行流；
- 内核保存当前任务现场；
- 选择另一个可运行任务；
- 恢复它的现场后继续跑。

这就制造了并发幻觉。

### 2.2 上下文切换本质是“保存现场 + 恢复现场”
现场一般包括：
- 通用寄存器；
- 程序计数器（PC）；
- 栈指针（SP）；
- 程序状态寄存器；
- 必要时还有浮点寄存器。

### 2.3 RTOS 的强大之处在于“主动让权”
裸机里你写 `delay()`，CPU 就是在原地空转。
RTOS 里如果任务进入：
- 延时等待；
- 等信号量；
- 等互斥锁；
- 等消息队列；

它就会被放进阻塞态，不再占用 CPU，处理器可以去跑别的任务。

## 3. RTOS 的调度核心

### 3.1 任务是什么
任务可以理解为：
- 一段独立的执行逻辑；
- 自己有独立的栈；
- 被调度器管理其状态。

常见状态有：
- Ready：准备好可运行；
- Running：当前正在运行；
- Blocked：等待事件；
- Suspended：被挂起。

### 3.2 为什么每个任务都要有自己的栈
因为函数调用链、局部变量、返回地址、保存寄存器都依赖栈。
如果多个任务共用一个栈，那么一切现场都会互相覆盖，根本不可能切换回来。

所以 RTOS 的任务切换，本质上也是在切换：

> **当前活跃的栈是谁。**

### 3.3 调度器做了什么
调度器核心工作可以概括为三步：
1. 维护可运行任务集合；
2. 决定下一个该跑谁；
3. 触发上下文切换。

不同 RTOS 策略可能不同，但常见 MCU RTOS 更偏向：
- 固定优先级；
- 高优先级抢占低优先级；
- 同优先级任务时间片轮转。

## 4. Cortex-M 中的 SysTick 与 PendSV

### 4.1 SysTick 是“时钟节拍源”
SysTick 通常周期性触发，例如每 1 ms 一次。它常用于：
- 维护系统 Tick；
- 判断任务延时是否到期；
- 决定是否需要调度。

### 4.2 PendSV 是“专门用来切任务的异常”
在 Cortex-M 里，PendSV 常被设计成最低优先级异常，专门负责上下文切换。

原因是：
- 它不应该打断更紧急的中断；
- 它适合在中断尾声统一完成“是否切任务”的决定。

### 4.3 一次典型切换的过程
常见过程如下：
1. SysTick 触发；
2. 内核更新 Tick，并检查是否有更高优先级任务就绪；
3. 若需要切换，则挂起 PendSV；
4. 进入 PendSV 后，把当前任务寄存器压栈；
5. 保存当前任务栈顶到 TCB；
6. 调度器选出下一个任务；
7. 从下一个任务的栈中恢复寄存器；
8. 异常返回后，CPU 继续执行新任务。

这就是“并发”在底层真正发生的动作。

## 5. 线程安全、竞态与同步原语

## 5.1 为什么会发生竞态条件
像下面这样的代码：
```c
counter++;
```
通常不是单指令完成，而是：
1. 读旧值；
2. 加 1；
3. 写回。

如果两个任务或中断同时操作，就可能丢失更新。

## 5.2 常见解决方法
### 互斥锁（Mutex）
适用于任务上下文之间保护共享资源。
特点：
- 允许阻塞；
- 能表达“只有一个任务能进入临界区”；
- 某些实现带优先级继承，防止优先级反转。

### 信号量（Semaphore）
更偏向“资源计数”或“事件通知”。

### 原子操作（Atomic）
适合非常短小的共享变量修改，比如计数器、标志位。

### 自旋锁（Spinlock）
适合极短临界区，等待时不睡眠而是忙等。常用于：
- 多核内核；
- 中断不能睡眠的上下文；
- 极短时间的硬件寄存器保护。

### 关中断
在单核 MCU 中非常常见，适合保护：
- 会同时被任务和 ISR 访问的共享数据；
- 极短临界区。

缺点是会影响中断响应实时性。

## 6. 为什么 Block 比裸机 Delay 高级

### 6.1 裸机 Delay
裸机里的延时通常是：
- while 循环空转；
- CPU 什么都干不了；
- 时间都浪费在等待里。

### 6.2 RTOS Delay
RTOS 的 `delay()` 通常不是让 CPU 忙等，而是：
- 把当前任务标记为 Blocked；
- 记录唤醒时间；
- 调度器去运行别的任务。

这就是 RTOS 能提升系统吞吐与响应性的根本原因之一。

### 6.3 等锁、等消息也是一样
任务拿不到锁或拿不到消息时，不该疯狂轮询，而应进入阻塞态。
这也是 RTOS 和简单 super loop 方案的本质差异之一。

## 7. 中断上下文与任务上下文

### 7.1 任务上下文
任务上下文允许的事情更多，例如：
- 可阻塞；
- 可等待信号量；
- 可进行较复杂的业务逻辑。

### 7.2 中断上下文
中断上下文特点是：
- 响应要快；
- 通常不允许阻塞；
- 不适合做太重的工作；
- 更适合“快速取数 + 标记事件 + 唤醒任务”。

很多工程问题都来自于：

> **把应该放在任务里的逻辑塞进中断。**

### 7.3 常见正确分工
- ISR：读取状态、清中断、把数据塞进缓冲区 / 队列、唤醒任务。
- 任务：做协议解析、状态机、业务逻辑、日志、界面更新。

## 8. Linux 里的线程与调度

虽然本篇重点是 RTOS，但可以顺带建立桥梁：
- Linux 线程本质上也是可调度实体；
- Linux 不只是简单优先级，而是有更复杂的调度策略；
- CFS 通过 `vruntime` 追求公平；
- 内核会维护更复杂的运行队列，而不是 MCU RTOS 常见的简单优先级表。

所以 RTOS 与 Linux 的底层思想并不割裂，核心仍然是：
- 保存上下文；
- 选择下一个执行流；
- 恢复上下文。

只是 Linux 做得更复杂、更通用。

---

## 9. 本专题最容易踩坑的点
- 把“多任务并发”误以为真的物理同时执行。
- 不理解“每个任务必须有独立栈”。
- 不知道 PendSV 和 SysTick 的角色分工。
- 在 ISR 里做复杂逻辑或阻塞操作。
- 把自旋锁、互斥锁、原子操作混着用。
- 忽视优先级反转和临界区过长问题。
- 用忙等代替阻塞，白白浪费 CPU。

# Linux 中的 sysfs、procfs、uevent 与 udevd 流程

## 1. 这篇文档解决什么问题

本专题回答的是：

> **Linux 内核里的设备、状态变化和用户空间到底是如何连起来的？为什么有时候看一个设备要去 `/sys`，有时候看系统状态又要去 `/proc`，而设备热插拔时又会冒出 `uevent` 和 `udevd`？**

很多人对这些名词都见过，但常常是分散理解：
- 知道 `/sys` 里有东西，但不知道它映射的是哪类内核对象；
- 知道 `/proc` 能看进程和中断，但不知道它和设备模型的边界；
- 知道 `udevd` 会创建设备节点，但不知道内核是怎么把“设备变化”通知到用户空间的。

这篇文档就是把这条链串起来。

---

## 2. 核心结论先看

### 2.1 `sysfs` 更偏“设备模型与内核对象的属性视图”
它通常用于展示：
- 设备
- 驱动
- bus
- class
- kobject 属性
- 一部分可调参数

也就是说，`sysfs` 更像是：

> **把内核对象树映射成文件系统。**

### 2.2 `procfs` 更偏“进程与系统运行状态视图”
它更常用于展示：
- 进程信息
- 内核统计信息
- 中断分布
- 内存信息
- 调试状态

所以 `/proc` 的重点是：

> **系统此刻在怎么运行。**

而不是“这块设备对象在设备模型里长什么样”。

### 2.3 `uevent` 是内核发往用户空间的异步通知
当设备状态变化、设备创建、移除、属性变化时，内核可以发出 `uevent`。
这相当于说：
- 内核捕获到了某件值得告诉用户空间的事；
- 然后通过事件机制把消息送出去。

### 2.4 `udevd` 是监听这些事件并执行用户空间动作的守护进程
它会根据规则：
- 创建设备节点；
- 设置权限；
- 建符号链接；
- 触发后续用户空间动作。

所以这条链可以先记成：

> **内核对象 → sysfs 展示 / 属性变化 → uevent 发消息 → udevd 响应并处理用户空间动作**

---

## 3. 为什么 Linux 需要多种“文件系统视图”

Linux 很喜欢把很多东西抽象成文件，但不同“文件系统视图”的目标不同。

### 3.1 为什么不把所有东西都塞到一个目录里
因为系统里至少存在两类完全不同的信息：
- **对象关系与属性**：某个设备是谁、挂在哪个 bus、属于哪个驱动、有哪些属性。
- **运行态统计与状态**：当前进程、CPU、中断、内存、内核状态。

这两类信息虽然都能以文件形式暴露，但语义差别很大。
所以 Linux 才分出：
- `sysfs`
- `procfs`
- 以及其他如 `debugfs` 等不同视图。

---

## 4. `sysfs`：设备模型的窗口

## 4.1 `sysfs` 映射的核心对象是什么
`sysfs` 背后最关键的概念是 **kobject**。
你可以粗略理解成：
- 内核里很多可组织、可引用的对象；
- 会挂进一棵层次树；
- 带有属性和生命周期。

而 `sysfs` 就是把这棵对象树的一部分暴露成目录和文件。

### 4.2 典型能在 `/sys` 看到什么
例如：
- `/sys/bus/`
- `/sys/devices/`
- `/sys/class/`
- `/sys/module/`
- `/sys/devices/system/cpu/`

这些路径背后不是普通配置文件，而是在反映：
- 总线关系
- 设备实例
- 设备类别
- 模块
- 系统对象

### 4.3 为什么很多驱动会在 `sysfs` 暴露属性
因为某些配置或状态适合用简单属性文件表达，例如：
- 当前模式
- 使能状态
- 阈值
- 调试计数
- 固件版本

但要注意，`sysfs` 适合的是：
- 简单属性
- 小而明确的控制语义

不适合复杂事务接口。

---

## 5. `procfs`：运行态和调试视角

## 5.1 `/proc` 更像系统仪表盘
`procfs` 的典型内容包括：
- `/proc/cpuinfo`
- `/proc/meminfo`
- `/proc/interrupts`
- `/proc/<pid>/`
- `/proc/cmdline`

这些文件更像在回答：
- 系统现在在怎样运行；
- 某个进程当前状态如何；
- 内存和中断怎么分布；
- 启动参数是什么。

### 5.2 为什么 `/proc/interrupts` 很重要
你之前关心 IRQ 亲和性和中断性能时，这个文件尤其有用，因为它能帮助你观察：
- 哪些中断在频繁发生；
- 落在哪个 CPU 核；
- 是否存在分布不合理或单核压力过高的问题。

### 5.3 `procfs` 和 `sysfs` 的直觉区别
一个很好记的区分方式是：
- `sysfs`：**对象和属性**
- `procfs`：**状态和运行**

虽然现实中会有边界灰度，但这个直觉非常好用。

---

## 6. 从设备创建到 `/dev` 节点：中间发生了什么

很多人以为：
- 驱动一注册，`/dev/xxx` 就自动出现。

其实中间还有完整链路。

### 6.1 内核侧先有设备对象
先得有：
- 设备被识别 / 注册；
- 驱动完成 probe；
- 设备号和 class 等关系建立。

### 6.2 内核产生用户空间可感知事件
当设备状态变化时，内核可以发出 `uevent`，里面会带上一些环境信息，例如：
- 这是 add、remove 还是 change；
- 设备路径是什么；
- 属于哪个 subsystem；
- 设备名或 devnum 是什么。

### 6.3 `udevd` 收到后再决定用户空间动作
它会依据规则：
- 是否在 `/dev` 下创建设备节点；
- 节点叫什么名字；
- 权限和组是什么；
- 是否创建符号链接；
- 是否触发脚本或其他动作。

所以 `/dev` 节点并不是“内核直接吐出来的普通文件”，而是：

> **内核事件 + 用户空间守护进程规则共同完成的结果。**

---

## 7. `uevent` 到底是什么

## 7.1 它的本质
`uevent` 可以理解成：
- 一种从内核到用户空间的事件广播机制；
- 通常基于 Netlink 把消息送到监听者。

### 7.2 它通常在什么时候发
常见场景包括：
- 设备添加；
- 设备移除；
- 媒体插拔；
- 状态变化；
- 某些驱动主动触发 change 通知。

### 7.3 为什么这很重要
因为用户空间很多联动都依赖它，例如：
- 插 U 盘时自动出现块设备；
- 某个输入设备插入后产生节点；
- 某个硬件状态变化后触发上层服务更新。

这就是你之前提到的“内核与 UI / 用户空间联动”的底层桥梁之一。

---

## 8. `udevd` 在做什么

## 8.1 它不是内核的一部分
`udevd` 是用户空间进程。它的意义是：
- 把内核事件转成用户空间可控动作；
- 让策略配置留在用户空间，而不是写死在内核里。

### 8.2 典型动作
- 创建设备节点；
- 改权限；
- 创别名 / 符号链接；
- 给不同类型设备分配不同规则；
- 配合桌面 / 服务层做更高层联动。

### 8.3 为什么这样设计更好
因为：
- 内核负责发现和通知；
- 用户空间负责策略和定制；

这样边界更清楚，也更灵活。

---

## 9. 一个完整例子：USB 串口插入时发生什么

可以用一个常见例子串起来：

1. 硬件插入，内核识别到新设备。
2. 对应 bus / driver 完成设备创建和匹配。
3. 驱动注册字符设备等内核对象。
4. 内核发出 `uevent`，说明某个设备 add 了。
5. `udevd` 收到事件，根据规则创建设备节点。
6. `/dev/ttyUSB0` 或类似节点出现。
7. 用户空间应用开始可通过该节点访问设备。

这条链里：
- `/sys` 能帮你看设备对象和归属；
- `/proc` 可能帮你看驱动运行状态和中断变化；
- `uevent` / `udevd` 负责把变化传给用户空间。

## 10. 驱动开发时，这些接口分别该怎么用

## 10.1 什么时候更适合 `sysfs`
适合：
- 暴露简单状态值；
- 暴露少量控制开关；
- 让用户空间以“属性文件”方式查看 / 修改设备状态。

不适合：
- 大块数据交换；
- 复杂事务流程；
- 性能敏感连续 IO。

## 10.2 什么时候看 `procfs`
适合：
- 系统级调试；
- 看中断、内存、进程、统计信息；
- 做运行时诊断。

## 10.3 什么时候依赖 `uevent`
适合：
- 设备状态变化需要通知用户空间；
- 需要触发用户空间动态响应；
- 热插拔或状态变更场景。

---

## 11. 一条非常好记的理解链

你可以把它记成：

> **设备模型存在于内核对象树里**  
> → `sysfs` 把对象和属性展示出来  
> → `procfs` 提供运行态和系统态观察视角  
> → 设备变化时内核发 `uevent`  
> → `udevd` 接住事件并完成用户空间设备节点和策略动作

这条链一旦通了，Linux 设备相关很多“为什么这样设计”的问题都会更容易理解。

---

## 12. 最容易踩坑的点
- 把 `sysfs` 和 `procfs` 当成两套随便选的查看方式。
- 认为 `/dev` 节点完全由内核自动直接创建。
- 不理解 `uevent` 是“通知”，而不是设备模型本体。
- 在不适合的场景滥用 `sysfs` 做复杂接口。
- 调试设备绑定问题时，只看 `/dev`，不看 `/sys` 和 `dmesg`。

---

## 13. 调试时的一个实用顺序

如果你遇到“驱动好像在，但用户空间看不到设备”这类问题，可以按这个顺序查：
1. 驱动 probe 是否成功。
2. `/sys` 中是否已出现对应设备对象。
3. 设备是否已归到预期 bus / class / driver。
4. 内核是否发出了相应 `uevent`。
5. `udevd` 规则是否生效。
6. `/dev` 节点是否被正确创建。

这个顺序通常能快速定位问题落在哪一层。

# Linux 驱动模型与用户 / 内核边界

## 1. 这篇文档解决什么问题

本专题回答两个核心问题：

> **Linux 是如何把硬件组织成统一驱动模型的？**  
> **用户程序又是如何跨过权限边界与内核交互的？**

这也是从 MCU / RTOS 进一步走向 Linux 驱动开发时，最容易出现认知断层的地方。

---

## 2. 核心结论先看

### 2.1 Linux 不是“直接调驱动函数”
用户态 App 不能像裸机那样直接访问寄存器或调用内核内部函数。通常要经过：
- 系统调用；
- 文件接口；
- `sysfs` / `procfs`；
- Netlink / `uevent`；
- 设备节点与驱动回调。

### 2.2 内核和用户态是两套不同规则的世界
- 用户态运行在受限权限级；
- 内核态拥有更高特权；
- 二者地址空间、可访问对象、可调用接口都不同。

### 2.3 驱动模型的目标是“解耦”
Linux 通过总线、设备、驱动三类对象，把“硬件描述”和“驱动实现”解耦：
- 设备告诉内核“我是什么”；
- 驱动告诉内核“我支持谁”；
- 匹配机制把两者绑定起来。

---

## 3. Bus / Device / Driver 三张表

### 3.1 Bus（总线）
Bus 不只是物理总线的意思，它更像一种组织框架。
常见例子：
- I2C bus
- SPI bus
- PCI bus
- platform bus

Bus 负责提供一套规则，说明设备和驱动该如何相遇、匹配和管理。

### 3.2 Device（设备）
Device 表示“某个硬件实例”。它包含的信息可能有：
- 名称；
- 地址；
- 中断号；
- DMA 资源；
- compatible 字符串；
- 所属 bus。

### 3.3 Driver（驱动）
Driver 表示“某种设备的支持实现”。它通常提供：
- probe
- remove
- suspend / resume
- 文件操作回调
- 中断处理
- DMA / 电源管理 / 时钟管理等逻辑

### 3.4 匹配机制
匹配本质上是：
- 设备声明自己是谁；
- 驱动声明自己支持谁；
- 内核按照总线规则把两者配对。

匹配方式可能基于：
- 设备 ID；
- compatible 字符串；
- ACPI / OF 表；
- 其他总线特有机制。

这就是你问到的“统一接口、启动时 Probe、运行时绑定”的核心闭环。

---

## 4. 为什么 platform_device 存在

MCU / SoC 内部很多外设不是插拔式设备，不会像 USB / PCI 那样自我枚举。
例如：
- GPIO 控制器；
- UART 控制器；
- I2C 控制器；
- 定时器；
- 看门狗；
- 时钟控制模块。

这类设备明明真实存在，但它们不会自己“告诉系统我在这”。
于是 Linux 需要一种虚拟总线框架来托管它们，这就是 `platform_bus` / `platform_device` 的意义。

换句话说：

> **platform 设备是对 SoC 内部固定硬件的一种软件建模方式。**

---

## 5. 设备树（Device Tree）到底解决什么问题

### 5.1 设备树的核心价值
设备树不是驱动代码，而是硬件描述数据。
它把这些内容从驱动里拆出来：
- 寄存器基地址；
- 中断号；
- DMA 通道；
- 时钟；
- GPIO；
- 电源属性；
- compatible 字符串。

### 5.2 为什么这很重要
如果没有设备树，驱动往往会写死板级参数：
- 可移植性差；
- 同一驱动适配多块板子很困难；
- 板级差异和驱动逻辑混在一起。

有了设备树后：
- 驱动代码只关心“怎么驱动”；
- 设备树负责“这个硬件具体在哪、用什么资源”。

这就是典型的数据与逻辑分离。

## 6. `sysfs`、`procfs`、`uevent` 与 `udevd`

## 6.1 `sysfs`
`sysfs` 更偏向展示：
- 设备模型对象；
- 内核属性；
- 驱动关联关系；
- 用户态可查看 / 可配置接口。

它本质上是把一部分内核对象和属性映射成文件系统视图。

## 6.2 `procfs`
`procfs` 更偏向：
- 进程信息；
- 内核运行状态；
- 调试与统计视图。

例如 `/proc/interrupts` 非常适合理解中断分布与亲和性。

## 6.3 `uevent`
`uevent` 本质是：
- 内核状态变化时向用户空间发出的异步通知；
- 常通过 Netlink 传递。

例如设备热插拔、状态变化等，都可以通过这种机制往用户空间发消息。

## 6.4 `udevd`
`udevd` 是用户空间守护进程。它会：
- 监听这些事件；
- 根据规则做动作；
- 创建设备节点；
- 触发用户空间联动。

这就是“硬件变化 → 内核捕获 → 用户空间处理”的典型闭环。

---

## 7. 用户态怎么进入内核态

### 7.1 系统调用路径
用户程序调用 `write()`、`open()`、`ioctl()` 这类接口时，并不是直接跳进普通 C 函数，而是：
- 触发特殊指令 / Trap / SVC；
- CPU 切换到更高权限级；
- 进入内核系统调用入口；
- 根据系统调用号转到对应处理函数。

### 7.2 为什么必须这样做
因为用户态不能随便：
- 访问内核数据结构；
- 修改设备寄存器；
- 控制中断；
- 操作页表或调度器。

这是一种安全与稳定性的强制隔离。

---

## 8. 为什么内核不能直接信任用户指针

用户态传进来的指针可能：
- 指向非法地址；
- 指向未映射页；
- 指向只读区域；
- 指向恶意构造的边界。

所以内核一般不能直接把用户指针当成可信内存去解引用，而要通过：
- `copy_from_user()`
- `copy_to_user()`

这些接口去做：
- 地址合法性检查；
- 权限检查；
- 安全复制。

这既是功能要求，也是安全边界要求。

## 9. MMU、页表与内核 / 用户地址空间

### 9.1 虚拟地址空间不是物理内存本体
程序看到的是虚拟地址，而不是直接看到物理 DDR。
真正的映射由 MMU + 页表完成。

### 9.2 用户空间与内核空间为什么能隔离
因为页表可以规定：
- 哪些虚拟页存在；
- 它们映射到哪些物理页；
- 谁能读、谁能写、谁能执行。

所以“用户区”和“内核区”的分离，本质上发生在虚拟地址与权限管理层面。

### 9.3 虚拟连续不代表物理连续
连续的虚拟地址，底层可能映射到零散物理页。
这也是为什么某些硬件场景会特别强调：
- 是否需要物理连续内存；
- 是否适合 DMA；
- cache 属性是否正确。

## 10. `malloc`、`calloc`、`kzalloc` 的语义差别

### 用户态
- `malloc`：分配，不清零。
- `calloc`：分配并清零。

### 内核态
- `kzalloc`：分配内核内存并清零。

### 为什么 `kzalloc` 清零很重要
它不只是“省得手动 memset”，还意味着：
- 防止旧数据残留被新对象看到；
- 降低未初始化字段导致的隐患；
- 有助于内核对象安全隔离。

### 为什么还要区分 `GFP_KERNEL` 和 `GFP_ATOMIC`
因为上下文不同：
- 进程上下文通常允许睡眠，可用 `GFP_KERNEL`；
- 中断上下文不能睡眠，必须用 `GFP_ATOMIC` 等不阻塞分配方式。

这和 RTOS / 中断上下文不允许阻塞的原则是相通的。

## 11. Linux 调度器与 IRQ 亲和性

虽然本篇重点不是调度器，但你问到的两个点很关键：

### 11.1 CFS 的核心思想
Linux 的普通任务调度并不是简单固定优先级，而是尽量追求公平。它会根据 `vruntime` 等指标组织可运行实体，让“被少运行的任务更容易得到 CPU”。

### 11.2 IRQ Affinity 的意义
在多核系统里，可以把某些中断偏向某个 CPU 核心处理。
这样做的意义包括：
- 降低核心间迁移；
- 改善 cache 命中；
- 避免高负载任务被无关中断打扰；
- 更容易做性能隔离。

---

## 12. 本专题最容易踩坑的点
- 以为用户态可以像裸机一样直接摸寄存器。
- 不理解设备树和驱动代码的职责分离。
- 把 `sysfs` 和 `procfs` 混为一谈。
- 不知道 `uevent` 是内核到用户态的通知链路。
- 直接信任用户指针而忽略边界检查。
- 忽视上下文差异，在错误上下文里做可能睡眠的操作。
- 把“虚拟地址连续”误当成“物理地址连续”。

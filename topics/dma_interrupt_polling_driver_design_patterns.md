# DMA、Interrupt、Polling 三种驱动设计模式

## 1. 这篇文档解决什么问题

本专题回答的是：

> **驱动里常见的轮询、中断、DMA 三种收发 / 控制模式到底该怎么选？它们是互斥关系，还是可以叠加使用？工程上为什么同一个外设在不同场景下会选不同模式？**

这是驱动设计里非常核心的决策问题。很多人会用 API，但不一定真正知道：
- 为什么某些场景适合 polling；
- 为什么某些场景中断更划算；
- 为什么高速收发几乎都会引入 DMA；
- 为什么 DMA 仍然离不开中断。

这篇文档就是把三种模式放到一个统一框架里比较。

---

## 2. 核心结论先看

### 2.1 三种模式本质上是在回答同一个问题
它们都在回答：

> **CPU 要以什么方式知道硬件准备好了，以及数据要由谁来搬。**

- **Polling**：CPU 自己反复去问“好了没”。
- **Interrupt**：硬件好了之后主动通知 CPU。
- **DMA**：硬件 / DMA 硬件自己搬数据，CPU 只在关键节点被通知。

### 2.2 它们不是绝对互斥的
真实工程里很常见的组合是：
- 配置阶段用 polling；
- 数据收发阶段用中断；
- 大块数据阶段用 DMA；
- DMA 完成后再用中断通知 CPU。

所以更准确地说，它们是：
- 三种不同层次的工作分担方式；
- 可以组合，而不是只能三选一。

### 2.3 选型的关键维度不是“高级不高级”，而是场景匹配
最重要的问题包括：
- 数据量大不大；
- 频率高不高；
- 时延要求严不严；
- CPU 宝不宝贵；
- 是否能接受阻塞；
- 数据是否适合批量搬运。

---

## 3. Polling：最直接，但最吃 CPU

## 3.1 它的本质
Polling 就是：
- CPU 主动不停读某个状态位；
- 看看外设有没有完成、有没有数据、能不能继续；
- 一旦满足条件，再继续下一步。

最直白的形式就是：
```c
while (!(REG_STATUS & READY_BIT)) {
    ;
}
```

### 3.2 它的优点
- 逻辑简单；
- 易理解；
- 不依赖中断系统；
- 在非常短小、非常确定的等待场景里，开销可能反而最低。

### 3.3 它的缺点
- CPU 被一直占着；
- 等待期间做不了别的事；
- 不适合高并发和低功耗；
- 容易因为忙等把系统实时性拖坏。

---

## 4. 什么时候 Polling 反而是合理的

很多人一听 polling 就觉得“太低级”。其实不对。

### 合理场景
- 启动阶段短时间等硬件 ready；
- 一次性初始化配置；
- 极少量数据、极短事务；
- 裸机简单系统；
- 禁中断环境下的短等待。

### 一个很重要的直觉
如果等待时间非常短，而：
- 配中断的开销更大；
- 切上下文反而不划算；

那 polling 可能就是最合适的。

所以它不是“落后”，而是：

> **在短小、可控、低复杂度场景下性价比很高。**

---

## 5. Interrupt：让硬件在准备好时叫 CPU

## 5.1 它的本质
中断模式下，CPU 不再一直盯着状态位，而是：
- 把外设配置好；
- 允许外设在事件发生时发中断；
- CPU 先去做别的事；
- 事件来时再进 ISR 处理。

### 典型适合的事件
- UART 收到一个字节 / 一帧；
- 定时器到期；
- GPIO 边沿变化；
- I2C / SPI 事务完成；
- DMA 完成或错误。

### 它的优点
- CPU 不用空转等待；
- 更适合事件驱动系统；
- 响应及时；
- 对中等频率、小块数据场景很合适。

### 它的缺点
- 中断频繁时会打碎 CPU 时间；
- ISR 写不好容易引入复杂并发问题；
- 每个小事件都打中断时，系统开销可能很高。

---

## 6. DMA：把“搬数据”交给硬件

## 6.1 它的本质
DMA 解决的是另一个层次的问题：
- 数据不要让 CPU 一个字一个字搬；
- 交给 DMA 控制器自己搬；
- CPU 只做配置和收尾。

### 它特别适合什么
- 大块数据；
- 高速收发；
- 持续流式数据；
- CPU 不想被搬运操作压垮的场景。

例如：
- UART 大包收发；
- SPI 屏幕刷图；
- ADC 连续采样；
- 音频流；
- 摄像头 / 高速接口数据流。

### 它的优点
- 大幅减少 CPU 搬运负担；
- 吞吐更高；
- 更适合持续高频数据流。

### 它的缺点
- 配置更复杂；
- 对缓冲区规划要求更高；
- 常牵涉 cache 一致性、对齐、环形缓冲、完成通知等问题。

---

## 7. 为什么 DMA 仍然常常要配合中断

这是一个很关键的误区澄清。

很多人以为：
- 用了 DMA 就不要中断了。

其实往往不是。

### 原因
DMA 只是负责搬运数据，但 CPU 还需要知道：
- 搬完了没有；
- 搬到一半没有；
- 出错没有；
- 下一块缓冲要不要接上。

所以常见模式是：
- DMA 干搬运；
- 中断做完成通知和流程接力。

也就是说：

> **DMA 不是中断的替代品，而是 CPU 搬运工作的替代品。**

---

## 8. 三种模式横向比较

## 8.1 CPU 占用
- Polling：最高
- Interrupt：中等
- DMA：最低（数据搬运角度）

## 8.2 实现复杂度
- Polling：最低
- Interrupt：中等
- DMA：最高

## 8.3 适合数据规模
- Polling：极小、极短事务
- Interrupt：小到中等事件 / 数据块
- DMA：中到大、持续型数据流

## 8.4 典型风险
- Polling：浪费 CPU、卡住系统
- Interrupt：中断风暴、并发复杂度
- DMA：一致性、缓冲管理、调试困难

---

## 9. 一些非常典型的设计模式

## 9.1 模式 A：初始化 polling，运行期 interrupt
很常见于：
- 启动时短暂等待 ready；
- 之后正常收发靠中断。

### 为什么合理
初始化阶段流程简单、总量小、对系统并发要求低，polling 很方便。
运行阶段则不想一直卡 CPU，所以改用中断。

## 9.2 模式 B：控制面 interrupt，数据面 DMA
这是很经典的一种模式。

例如：
- CPU 配置外设；
- DMA 搬大块数据；
- 完成后中断通知 CPU 收尾。

### 为什么这是工程上最常见的高性能方案
因为它把三件事分工得很清楚：
- CPU：做决策；
- DMA：做搬运；
- 中断：做事件通知。

## 9.3 模式 C：短包 interrupt，大包 DMA
例如串口 / SPI / 网络等经常会这样分：
- 小控制包直接中断收发；
- 大数据包切到 DMA。

### 为什么这样做
因为 DMA 配置不是零成本。
对于很小的数据，DMA 的 setup 开销可能比直接中断处理还不划算。

---

## 10. 选型时最实用的决策顺序

你可以按这几个问题来选：

### 问题 1：等待时间短不短
- 极短：先考虑 polling 是否最简单。

### 问题 2：事件频率高不高
- 不高：中断通常很合适。
- 很高：要警惕中断风暴。

### 问题 3：数据块大不大
- 小：中断就够。
- 大：优先考虑 DMA。

### 问题 4：CPU 是否还有更重要的事
- 如果 CPU 很紧张，不要让它当搬运工。

### 问题 5：系统是否实时敏感
- 太多 polling 会拖延响应。
- 太多中断会打碎调度。
- DMA 则需要处理好完成时机和缓冲一致性。

---

## 11. 不同外设中的典型选择直觉

## UART
- 少量调试日志：polling 也常见。
- 普通收发：interrupt 很常见。
- 大量连续数据：DMA 更合适。

## SPI
- 短寄存器配置：polling / interrupt 都可。
- 大块屏幕刷图 / Flash 数据搬运：DMA 常见。

## I2C
- 配寄存器、读小量状态：polling 或 interrupt 常见。
- 连续大块收发：某些平台会配 DMA。

## ADC
- 单次采样：polling / interrupt 可行。
- 连续采样流：DMA 极常见。

---

## 12. 最容易踩坑的点
- 把 polling 一概视为“低级方案”。
- 把中断一概视为“更高级，所以总该用”。
- 用 DMA 后忘了处理完成通知和一致性。
- 对很小数据也强行上 DMA，结果复杂度升了收益很小。
- ISR 里做太多，导致本来想节省 CPU，结果中断开销更大。
- 不把模式分成“控制面”和“数据面”思考。

---

## 13. 一个非常好记的总结

你可以把三者记成：
- **Polling**：我自己一直盯着看。
- **Interrupt**：好了你叫我。
- **DMA**：搬运你们自己去做，完了再叫我。

这三句话几乎能解释绝大多数驱动设计模式差异。

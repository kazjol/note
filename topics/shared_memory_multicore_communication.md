# 共享内存与多核 / 异构核通信

## 1. 这篇文档解决什么问题

本专题回答的是：

> **大核和小核、CPU 和 MCU、两个执行实体之间，如果它们都能访问同一块 SRAM，那么“共享内存通信”到底是怎么成立的？为什么改某几个地址或某一位就能让另一边执行动作？这里面真正靠的是什么机制？**

这正对应你前面最核心的一类问题：
- 大 CPU 和小 MCU 通过共享 SRAM 是怎么下命令的；
- 为什么这看起来像“改个 bit 就触发事情”，但本质又不是硬件魔法；
- 多核并发里为什么会有数据竞争、轮询、信号量、缓存一致性等一整串问题。

---

## 2. 核心结论先看

### 2.1 共享内存通信的本质不是“内存自己会通信”
共享内存只是：
- 一块双方都能访问的存储区域；
- 真正让它成为“通信机制”的，是双方**预先约定好的数据格式和时序规则**。

也就是说：

> **共享内存本身只是白板，协议约定才是语言。**

### 2.2 改一个 bit 之所以有意义，是因为另一边在按约定解释它
例如：
- bit0 表示“新命令到达”；
- 某地址偏移表示命令号；
- 某段内存表示 payload；
- 某个状态字表示 ACK / busy / done / error。

如果没有另一边的程序持续按这个协议解析，这些地址和位本身没有任何“神奇功能”。

### 2.3 共享内存通信真正难的地方，不在“能不能写进去”，而在“一致性、顺序、并发和通知”
常见难点包括：
- 谁先写，谁后写；
- 什么时候另一边能看见；
- Cache 是否已经回写；
- 是否会 RMW 冲突；
- 轮询还是中断通知；
- 谁来清状态位；
- 掉电 / 复位后共享区怎么恢复。

---

## 3. 共享内存通信最小模型

你可以先把它想成一个非常简单的邮箱系统。

例如共享区里有：
- `cmd`
- `arg0`
- `arg1`
- `status`
- `flag`

一边（发送方）做：
1. 把参数写好；
2. 写入命令号；
3. 最后把 `flag` 置位，表示“新命令到了”。

另一边（接收方）做：
1. 轮询 `flag`；
2. 看到置位后读取 `cmd` 和参数；
3. 执行命令；
4. 写回 `status`；
5. 清除 `flag` 或设置完成位。

### 核心理解
这看起来像“改一个 flag 就完成通信”，但真实意义是：
- 前面的数据区已经准备好；
- `flag` 只是“现在可以按协议解释这块内存了”的通知信号。

---

## 4. 为什么共享内存比“直接发协议包”更底层

和 I2C / SPI / UART 相比，共享内存通信通常更贴近本地存储访问：
- 双方不用在外部总线上逐 bit 发送帧；
- 而是都直接访问同一块物理存储；
- 通信成本更像“写表格 + 看表格”。

### 但为什么它又不是纯 MMIO
如果这块区域只是普通 SRAM，那么它不是外设寄存器意义上的 MMIO。
它只是：
- 一块普通内存；
- 被双方当作通信缓冲区使用。

所以它在“物理访问方式”上像普通内存，在“软件语义”上却承担了消息通道的角色。

---

## 5. 多核 / 异构核场景里常见的共享区结构

一个稍微像样的共享区常包含：
- 命令区
- 参数区
- 返回值区
- 状态区
- 环形缓冲 / 队列
- 心跳区
- 错误码区
- 版本 / 协议头

### 为什么要加版本和协议头
因为共享内存通信很容易演变成“双方都觉得自己懂这块内存”。
一旦版本升级、字段调整、长度改变，就可能出现：
- 老核按老格式读，新核按新格式写；
- 字段错位；
- 状态机完全错乱。

所以共享区其实也应该被当成一种“接口协议”来设计，而不是随便放几个全局变量。

---

## 6. 轮询为什么这么常见

## 6.1 轮询的优点
共享内存通信里，轮询非常常见，因为它：
- 简单；
- 容易跨核 / 跨系统实现；
- 不一定依赖额外中断线；
- 对小 MCU 或极简通信场景特别实用。

### 典型形式
接收方会周期性检查：
```c
if (shared->flag == 1) {
    // 读命令并执行
}
```

## 6.2 轮询的代价
- CPU 周期被消耗；
- 响应有轮询周期延迟；
- 对低功耗不友好；
- 高负载时会浪费带宽。

所以轮询简单，但并不是免费。

---

## 7. 什么时候会引入中断 / 门铃机制

在更高级一点的系统里，除了共享内存外，还会加一种“通知机制”，例如：
- 专用中断线；
- mailbox 外设；
- doorbell / event 寄存器；
- 硬件信号量事件；
- 核间中断（IPI）。

### 这样做的意义
共享内存负责装数据，通知机制负责说：
- “你现在该看了”。

这样就避免了纯轮询一直空转。

### 一句话理解
- **共享内存**：放消息内容。
- **中断 / 门铃**：告诉对方消息到了。

这两者合在一起，通信效率和实时性会好很多。

## 8. 数据竞争为什么会发生

这是共享内存最容易出事的地方之一。

### 场景 1：RMW 冲突
如果双方都在做类似：
```c
shared->flag |= (1U << 3);
```

底层通常是：
1. 读旧值；
2. 修改某位；
3. 写回。

如果两个执行流同时这样干，就可能互相覆盖。

### 场景 2：写入顺序问题
如果发送方先把 `flag` 置位，再慢一点写数据区，那么接收方可能：
- 先看到 `flag`；
- 但读到的是旧 payload。

所以必须明确：
- 先写数据；
- 最后写 flag；
- 接收端先看 flag，再读数据。

这其实就是最基本的“发布顺序”问题。

---

## 9. Cache 一致性为什么是共享内存通信的大坑

如果两个核都带 Cache，事情就复杂很多。

### 典型问题
发送方觉得自己已经写好了共享区，但其实：
- 数据还停在本核 Cache；
- 另一核去共享 SRAM 读时还看到旧值。

或者接收方一直读自己的旧 Cache 行，导致：
- 明明共享区已更新；
- 但它还像没看到一样。

### 解决思路
常见方法有：
- 共享区设为 Non-Cacheable；
- 发布前 clean cache；
- 接收前 invalidate cache；
- 用内存屏障保证顺序；
- 使用平台提供的 cache-coherent 共享机制。

这也是为什么很多“多核共享内存通信明明逻辑对但偶尔失效”的根因，其实在 cache，而不在协议本身。

---

## 10. 内存屏障和顺序为什么重要

共享内存通信里，不只是“有没有写进去”重要，“什么时候对别人可见、顺序是什么”也很重要。

### 一个典型发布流程
发送方应大致遵守：
1. 写 payload；
2. 做必要的内存屏障 / cache clean；
3. 最后写 ready flag；

接收方应大致遵守：
1. 观察 ready flag；
2. 做必要的内存同步 / cache invalidate；
3. 再读 payload。

### 为什么不能想当然
因为在现代 CPU / 总线 / cache 系统里：
- 写入可见顺序未必和代码顺序完全一样；
- 某些优化和缓存会让“你以为已经写完”不等于“别人已经能看见”。

---

## 11. 硬件信号量、Spinlock、位带这些东西在这里干嘛

## 11.1 硬件信号量（HSEM）
用于：
- 保证某段共享资源访问的独占性；
- 让两个核不会同时改同一块控制区。

### 典型用途
- 核 A 先拿 HSEM；
- 改共享结构；
- 释放 HSEM；
- 核 B 才能进入。

## 11.2 Spinlock
如果平台或软件栈提供核间自旋锁，也可用于保护极短临界区。

## 11.3 位带 / 原子位操作
某些架构允许对单 bit 做更安全的原子访问，这对共享 flag 特别有价值。

### 核心理解
这些机制不是在替代共享内存，而是在给共享内存协议加“并发秩序”。

## 12. 一个比“单 flag”更成熟的通信模型

更成熟的共享内存模型常常会做成环形队列：
- `write_idx`
- `read_idx`
- message slots
- 每个 slot 有长度、类型、payload、状态

### 为什么要这么做
因为真实系统里很快就会遇到：
- 不只一条消息；
- 发送比接收快；
- 需要批量消息；
- 需要不同类型命令并存；
- 需要避免单 mailbox 被覆盖。

所以共享内存通信最终常会从：
- “几个寄存器位”
发展成：
- “一个轻量级共享队列协议”。

---

## 13. 复位与异常恢复为什么麻烦

共享内存最大的工程坑之一是：
- 两边不一定同时复位；
- 一边重启后可能不知道共享区是干净还是脏的；
- 旧 flag、旧状态、旧索引都可能残留。

### 典型后果
- 新启动一侧误读旧命令；
- 队列索引错位；
- 双方都以为对方 busy；
- 心跳和状态字永久不一致。

### 预防办法
- 共享区加 magic / version / sequence；
- 启动时做握手；
- 明确哪一侧负责初始化共享区；
- 做超时和错误恢复；
- 心跳超时后允许重新同步。

---

## 14. 一个实用的设计原则

设计共享内存通信时，可以按这 5 个问题来规划：

1. **数据格式是什么？**
2. **发布顺序是什么？**
3. **通知机制是什么？**
4. **并发保护是什么？**
5. **异常恢复是什么？**

如果这 5 个问题没有写清楚，后面几乎一定会出问题。

---

## 15. 一条非常好记的理解链

你可以把共享内存通信记成：

> 共享 SRAM 提供共同可见空间  
> → 协议定义字段和状态位含义  
> → 发送方先写数据后发布标志  
> → 接收方通过轮询或中断获知更新  
> → 双方用锁 / 原子 / 屏障 / cache 同步保证一致性  
> → 出错时靠握手、心跳、超时和重同步恢复

这条链几乎能覆盖绝大多数多核共享内存通信设计。

---

## 16. 最容易踩坑的点
- 把共享内存误以为“只要双方能读写就自动形成通信”。
- 只设计数据区，不设计状态机和通知机制。
- 忽视先写数据再置 flag 的顺序约束。
- 忽视 cache 一致性。
- 用普通 RMW 去改共享标志，结果被并发覆盖。
- 没有复位后的重同步方案。
